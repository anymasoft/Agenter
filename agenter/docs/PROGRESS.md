## Открытые архитектурные вопросы (требуют решения)

Список вопросов, которые **не блокируют текущую разработку**, но должны быть проработаны и зафиксированы как ADR прежде чем мы упрёмся в них в реальной работе.

| № | Вопрос | Когда становится критичным | Заготовка ADR |
|---|---|---|---|
| AQ-1 | **Расширение vs основная конфигурация vs существующее расширение.** Сейчас агент по умолчанию работает в `ext_src/` — это правильно для CFE-разработки. Но не любая доработка должна жить в расширении: иногда нужно править основную конфигурацию (например для крупных рефакторингов от вендора), иногда — добавлять в уже существующее чужое расширение. Нужны критерии выбора + механизм управления. | Когда задача из ТЗ окажется НЕ под CFE (изменение типового справочника, удаление реквизита, рефакторинг кода вендора) | ADR-015 (статус: OPEN) |
| AQ-2 | **Снятие/расширение лимита turns при платном использовании (вариант D из ADR-016).** Текущая модель (B+E, гибрид) — компромисс для бесплатного личного использования: 120 эффективных turns + 5 ask_user, агент обязан укладываться. Когда появится своя система токенов / биллинг / BYOK — логика меняется: если юзер платит, он сам решает сколько турнов потратить. Тогда переход на вариант **D (полный сброс счётчика после каждого ask_user)** или вообще на «лимит = деньги» становится правильным. | Когда внедрим биллинг / собственную систему токенов / коммерческую версию (Фаза 7 PRO-edition: BYOK + бизнес-модель) | См. ADR-016 секция «Будущая эволюция» |

**AQ-1 — детализация (как решать):**
1. **Критерии выбора** (надо описать как чек-лист в системном промпте):
   - Добавление новых объектов с собственным префиксом → расширение
   - Изменение поведения типовых процедур через `&Перед/&После/&Вместо` → расширение
   - Структурное изменение типового объекта (удаление реквизита, смена типа) → **основная конфигурация** (CFE не умеет)
   - Изменение прав на типовой объект → **основная конфигурация** (роли в CFE ограничены)
   - Точечная правка отчёта/обработки клиента → выбор зависит от того, как клиент хочет обновляться
2. **Механизм управления**:
   - Опция A: ручной переключатель в UI (Сайдбар: «Режим: CFE / Конфигурация / Чужое расширение»)
   - Опция B: указание в промпте (`/mode:config`, `/mode:ext=<имя>`)
   - Опция C: агент сам **уточняет через `ask_user`**, если из контекста неясно (теперь это технически возможно после ADR-013)
   - Скорее всего нужна **комбинация**: дефолт + переопределение промптом + ask_user при сомнении
3. **Поддержка нескольких расширений**: текущий `ext_src_path` — один. Возможно нужен список расширений, и `meta_compile`/`cfe_borrow` будут спрашивать какое.
4. **Безопасность**: правка основной конфигурации — операция с **гораздо более высоким риском** (нет легковесного отката через `Удалить расширение`). Гард `db_load` для основной конфигурации должен требовать дополнительного подтверждения юзера.

**Когда решать**: после следующих 2-3 реальных прогонов больших ТЗ, когда станет видно, какие сценарии чаще встречаются (CFE-only, mixed, config-only). Преждевременная фиксация = риск спроектировать под несуществующую проблему.

**Реализация 📎 (2026-05-15)**: multi-file picker (txt/md/bsl/os/xml/json/csv/yaml/log/html/ini, до 2 МБ); чип с именем/размером и крестиком над composer-input; содержимое склеивается с текстом промпта блоками с разделителями `─` перед отправкой; в UI-сообщении показывается компактное резюме (имена файлов), полный текст уходит в LLM.

**Реализация ask_user (2026-05-15)**: см. ADR-013. Tool `mcp__agenter__ask_user(question, options?)` блокирующий, регистрируется только когда run_task получил callback `on_ask_user`. Backend: `pending_questions` registry в AppState + asyncio.Event; REST `POST /tasks/{id}/answer`; WS-события `ask_user` / `ask_user_resolved`. Frontend: `AskUserDialog` overlay с blur, radio (если options) или textarea (если нет), Ctrl+Enter для отправки, кнопка «Прервать задачу». Sentinels `[CANCELLED]` / `[TIMEOUT 10min]`. Q+A автоматически попадает в exec-log как markdown-блок и в БД задачи.

**Также 2026-05-15**:
- **Continuous sessions + Phases + Snapshots + MEMORY.md** (ADR-018) — **большой refactor под длинные задачи**:
  - Auto-resume Claude SDK сессии: `ClaudeAgentOptions(resume=sdk_session_id)`. Каждая новая POST /tasks продолжает разговор предыдущей задачи; TodoWrite, ToolResult'ы, прочитанные файлы — всё переходит между задачами. session_id ловится из `SystemMessage(subtype="init")`, сохраняется в `sessions(project_id, sdk_session_id, snapshot_path)`. Fallback: если resume не удался (session протухла) — graceful запуск новой
  - **Фазы и phased commits**: системный промпт обязывает агента делить задачи 5+ объектов на фазы с маркером `🔒` в TodoWrite. Каждый успешный `db_load` автоматически = «commit фазы N» (детект в PostToolUse). При BUDGET EXHAUSTED фазы 1..N-1 уже в БД, фаза N остаётся в ext_src
  - **Auto-snapshot ext_src/** при старте новой сессии: zip в `agenter/data/<project>/snapshots/<task_id>_<ts>.zip`, TTL 7 дней, лимит 5 snapshots/project. Откат через `POST /tasks/{id}/rollback` (rmtree ext_src → unzip)
  - **MEMORY.md как долговременная память**: `agenter/data/<project>/MEMORY.md` инлайнится в system_prompt автоматически (страховка от потери в SDK-истории). Агент сам обновляет через Edit/Write. Переживает reset сессии
  - **Честный UI**: убрано ложное «Изменения применены в 1С». TaskFinalStatus показывает реальное состояние: 🟢 applied (все фазы committed) / 🟡 partial (часть committed) / 🟡 staged (только ext_src, БД не тронута) / 🔴 failed. Кнопка «Откатить ext_src/» рядом
  - **SessionBadge** над composer'ом: «🔗 Контекст активен · N задач · snapshot ✓» + кнопка «🆕 Новая». Сброс через ConfirmDialog
  - **REST**: `GET/POST /sessions/{pid}`, `GET /tasks/{id}/phases`, `POST /tasks/{id}/rollback`. **Параллельные задачи в одном проекте → 409** (страховка от race condition)
  - SQLite: миграция через ALTER TABLE (idempotent), новые таблицы `sessions`, `task_phases`, новые поля в `tasks` (`session_id`, `project_id`, `phases_total/committed`, `db_load_count`, `final_state`)
  - WS-события: `session_updated`, `session_reset`, `phase_committed`, `phase_failed`, `task_state_changed`, `snapshot_created`, `snapshot_restored`
- **Переключатель модели + расход токенов** (ADR-017): popup в composer-bar (Sonnet 4.6 default / Opus 4.6 / Haiku 4.5), цена за 1M токенов рядом с названием, выбор сохраняется в localStorage. Backend: `ALLOWED_MODELS` + `MODEL_PRICING` в `app/main.py`, endpoint `GET /models`, `TaskRequest.model`, динамический `ClaudeAgentOptions(model=...)`. Per-turn usage в логах (мелкий серый стиль): `↑in ↓out · cache · ~$cost`, финальная «📊 Итого» с `num_turns` и `total_cost_usd` (из SDK или нашего расчёта). Auto-режим **намеренно не реализован** — попадёт в Фазу 6 PRO-edition вместе с настоящим multi-model routing
- **Раздельный бюджет turns** (ADR-016): `MAX_ITERATIONS = 120` эффективных tool calls + `MAX_ASK_USER_PER_TASK = 5` уточнений отдельным лимитом + `SDK_HARD_CAP_TURNS = 200` страховочный потолок. ask_user НЕ расходует эффективный бюджет — юзер не штрафует агента за уточнения. Своя пара счётчиков (`effective` / `ask_user`) в общем `turn_state` для pre+post hooks, на 80% эффективного бюджета лог-предупреждение, при исчерпании — PreToolUse блокирует все tools с просьбой завершить финальным ответом
- Динамический pre-flight кэш known_objects — см. ADR-014. После каждого успешного meta_compile/cfe_borrow/cfe_patch_method кэш пересобирается → ссылки на свежесозданные ext_src-объекты не блокируются ложно
- Системный промпт в `orchestrator_sdk._build_client_context`: добавлены секции «когда вызывать ask_user», «правило 3 повторов», «не используй Bash для ls/cat/find — это Glob/Grep/Read», «pre-flight обновляется автоматически», «бюджет turns и завершение задачи», «фазы и коммиты», «континуация сессии», «память проекта»

**Остаток (↑ + slash + @ + 🖼️ + удалить `<>`): ~7 часов работы.**
