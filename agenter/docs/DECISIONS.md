## ADR-019 · Без Docker — нативные tools вместо контейнеризированных MCP
**Дата:** 2026-05-16
**Контекст:**
В аудите 2026-05-15 было предложено подключить 4 external MCP-сервера для повышения качества работы агента: SyntaxCheckServer (BSL LS), HelpSearchServer (RAG по платформе), FormsServer (генерация форм), Graph_metadata_search (граф-поиск). Все они распространяются как Docker-образы (`comol/*` и `mcp-forms-server`). Пользователь явно сообщил что **не использует Docker и не хочет его ставить** на свою машину.

**Анализ возможности обойтись без Docker:**

| Сервер | Источник | Можно без Docker? |
|---|---|---|
| syntax-check | github.com/1c-syntax/bsl-language-server, MIT, open-source | ✅ **Да** — есть GraalVM-native .exe в релизах (не требует ни Docker, ни Java) |
| help-platform | RAG над shcntx_ru.hbk | ✅ **Уже есть нативно** — наш `platform_doc_lookup` поверх SQLite-индекса shcntx |
| ssl-search | comol/mcp_ssl_server, proprietary | ❌ Закрытый продукт. Частичная замена: `bsl_*` поиск по SCHEME, где БСП лежит как обычная конфа |
| forms-server | mcp-forms-server, proprietary build | ❌ Закрытый + custom build. Замена: 4 наших `form_*` tools + `skill_search` |
| graph-metadata | comol/1c_graph_metadata + Neo4j + OpenAI | ❌ Proprietary + двойная зависимость (Neo4j, OpenAI API key). Замена: BSL Atlas + `metadatasearch` |
| code-metadata | proprietary | ❌ Покрыт BSL Atlas (`codesearch`, `search_code_filtered`) |
| 1CCodeChecker | proprietary, платный | ❌ — |
| templates-search | proprietary | ❌ Замена: локальная папка шаблонов + `skill_search` |

**Решение:** для syntax-check добавляем нативную интеграцию upstream BSL LS (Sprint 2 S2.1, модуль `agenter/app/bsl_ls.py`). Остальные Docker-only сервера НЕ подключаем — у нас уже есть нативные эквиваленты (BSL Atlas + наш `platform_doc_lookup` + `form_*` + `meta_*` + `skill_search`) которые в сумме покрывают тот же usecase.

**Что это даёт:**
- Нулевая зависимость от Docker / WSL2 / Docker Desktop.
- Нулевая зависимость от Java (BSL LS — GraalVM native binary).
- Нулевая зависимость от внешних API (OpenAI для graph-metadata не нужен).
- Никаких лицензионных ключей от сторонних поставщиков.
- Все tools работают «из коробки» после установки одного .exe (для BSL LS) или вообще без установок (fallback в Python).

**Что мы теряем:**
- Качество автодополнения BSL не на 100% уровне настоящих коммерческих linter'ов (но BSL LS native — 95% качества).
- Граф-поиск по метаданным проигрывает Neo4j по «семантическим» запросам, но выигрывает по простоте и работает без OpenAI.
- БСП-справка ищется через bsl_* по самой БСП (если выгружена) — медленнее семантического RAG, но без подписки.

**Конфиг:**
Docker-only MCP сервера остаются в `agenter/config/config.json` с `enabled: false` и описанием `[DOCKER-ONLY, не используется]` — как документация о том что было рассмотрено и почему не подключено. Включать вручную никто не запрещает (агент готов к их wildcards в `_ALLOWED_TOOLS`), но default-конфигурация — без них.

**Связано:** ADR-018 (фазы), audit-2026-05-15 (полный план).

## ADR-018 · Continuous sessions + Phases + Snapshots + MEMORY.md
**Дата:** 2026-05-15
**Контекст:**
Большой Промпт.txt (агентские продажи, 9 шагов) выявил три критические дыры:
1. **Context loss между задачами** — каждая POST /tasks создавала новую SDK-сессию, TodoWrite и история разговора терялись. «Продолжай с шага 5» означало «начни с нуля».
2. **Транзакционность непродумана** — агент копил изменения в ext_src/ и звал db_load в самом конце задачи 9. При BUDGET EXHAUSTED после шага 4 — в БД ноль, ext_src в полу-состоянии.
3. **UI врал** — захардкоженное «Готово. Изменения применены в 1С» показывалось всегда при task_done, независимо от того был ли успешный db_load.

**Варианты (для context loss):**
- A. Auto-resume SDK сессии (через `ClaudeAgentOptions(resume=session_id)`)
- B. Кнопка «Продолжить» рядом с «Отправить»
- C. MEMORY.md + summary state file
- D. Гибрид A + C

**Варианты (для транзакционности):**
- A. Atomic — один db_load в конце всей задачи (даже многозадачной resume-цепочки)
- B. Phased commits — агент делит большие задачи на фазы (5-7 объектов), db_load в конце каждой
- C. Transactional staging — ext_src_staging/, commit явный

**Выбор:**
- Context loss: **D (гибрид A + C)** — auto-resume для прямой continuity + MEMORY.md как долговременная память переживающая сброс
- Транзакционность: **B (phased commits + snapshot + честный UI)**
- Сброс сессии: только по кнопке «🆕 Новая» (auto-cleanup не активен)
- Все вместе одним пакетом

**Обоснование:**
- Resume — нативный механизм SDK, минимум кода, полный контекст (включая ToolResult, размышления, TodoWrite). MEMORY.md страхует от потери при сбросе и работает как «знания о проекте» (префиксы, UUID, особенности).
- Phased commits решают «огромная задача оборвалась на полпути» — фазы 1..N-1 уже в БД, можно продолжить с N в новой задаче (через resume).
- Auto-snapshot ext_src/ при старте сессии даёт безопасный откат — БД и так нетронута до первого db_load, а файлы можно восстановить.
- Честный UI устраняет дезинформацию: applied / partial / staged / failed на основе подсчёта успешных db_load в логе задачи.

**Реализация:**
- SQLite: таблицы `sessions(project_id, sdk_session_id, snapshot_path, ...)`, `task_phases(task_id, phase_index, status, ...)`, новые поля в `tasks` (session_id, phases_total/committed, db_load_count, final_state)
- `agenter/app/snapshots.py` — zip/restore ext_src, TTL 7д + лимит 5 snapshots/project
- `orchestrator_sdk.run_task(..., resume_session_id, on_session_captured, on_phase_commit, memory_md)` — захват session_id из `SystemMessage(init)`, передача `resume=` в `ClaudeAgentOptions`, инлайн MEMORY.md в system_prompt, детект db_load → phase commit
- Системный промпт: новые разделы «ФАЗЫ И КОММИТЫ», «КОНТИНУАЦИЯ СЕССИИ», «ПАМЯТЬ ПРОЕКТА»
- REST: `GET /sessions/{pid}`, `POST /sessions/{pid}/reset`, `GET /tasks/{id}/phases`, `POST /tasks/{id}/rollback`. Параллельные задачи в одном проекте — 409 Conflict.
- WS-события: `session_updated`, `session_reset`, `phase_committed`, `phase_failed`, `task_state_changed`, `snapshot_created`, `snapshot_restored`
- UI: `SessionBadge` над composer'ом (🔗 активна + кнопка 🆕 Новая), `TaskFinalStatus` (4 варианта вместо ложного текста), `PhasePill` для фаз, `ConfirmDialog` для reset/rollback

**Подводные камни / решения:**
- **Resume протух** на стороне Anthropic → graceful fallback: ловим exception при подключении, начинаем новую сессию + предупреждение в UI
- **Параллельные задачи в одной сессии** → блокируем 409 на бэке (UI и так превращает кнопку в «Стоп»)
- **MEMORY.md разрастание** → инструкция в промпте «< 100 строк, делай compact», load с ограничением 50000 символов
- **Бюджет turns при resume** → `turn_state` создаётся в каждом `run_task` заново → юзер получает свежие 50 turns после продолжения. Намеренно: фактически это новая порция работы.
- **Snapshot не откатывает БД** → юзер сам решает через отдельный db_load после rollback. UI явно об этом предупреждает в confirm-диалоге.
- **Phase detection** автоматический (каждый успешный db_load = коммит фазы N), агент не размечает явно — проще для LLM.

**Связано с:**
- ADR-016 (бюджет turns) — turn_state работает per-run_task, не накапливается через resume
- ADR-013 (ask_user) — счётчик ask_user тоже per-task, при resume сбрасывается
- ADR-015 (extension vs main config, OPEN) — фазы помогают и здесь: разные фазы могут жить в разных местах
- AQ-2 (Phase 7: BYOK + биллинг) — после введения собственного биллинга, возможно полный сброс счётчиков (D-вариант ADR-016) станет дефолтом

**Файлы:**
- `agenter/app/main.py` — миграция SQLite, эндпойнты, wrapper, callbacks
- `agenter/app/orchestrator_sdk.py` — resume, session capture, phase detection, MEMORY.md инлайн, новые системные разделы
- `agenter/app/snapshots.py` — НОВЫЙ: zip/restore + MEMORY.md utils
- `agenter/backend/main.py` — расширение «ОБЩИЙ ПРИНЦИП» правилом про фазы
- `agenter/frontend/api.js` — getSession/resetSession/getTaskPhases/rollbackTask
- `agenter/frontend/agenter-connect.js` — WS handlers session_*/phase_*/snapshot_*
- `agenter/frontend/chat-screen.jsx` — SessionBadge, TaskFinalStatus, PhasePill, ConfirmDialog, state, useEffect
- `agenter/frontend/styles.css` — .session-badge, .task-status-*, .phase-pill-*, .rollback-btn

## ADR-001 · Использование MetadataViewer1C как основы для metadata_utils
**Дата:** 2026-05-14
**Контекст:** Нужен парсер XML-выгрузки конфигурации 1С для работы агента со структурой проекта. Писать с нуля — недели работы. Хотим переиспользовать готовое.

**Варианты:**
- A. Писать парсер с нуля (lxml + ad-hoc для каждого типа объектов)
- B. Использовать вscode-1c-metadata-viewer (~1162 строк, простой TreeView)
- C. Использовать MetadataViewer1C (~7700 строк, активный проект, React UI, TS-типы)

**Выбор:** C
**Обоснование:**
- Огромная экосистема: парсеры для Catalog/Document/Register/Enum/CharOfAccounts/..., универсальный сканер, JSON-схемы XDTO платформы
- 27 готовых JSON-схем XDTO (MDClasses, XcfLogForm, DcsSchema, ...) → можем валидировать любой XML 1С
- 47×2 SVG-иконки для UI
- Открыт под MIT/Apache (open-source).

**Что портировано:**
- `metadata_utils/unicode_name.py` — декодер `#UXXXX` (5 строк)
- `metadata_utils/metadata_scanner.py` — сканер выгрузки (2 формата)
- `metadata_utils/metadata_parser.py` — универсальный парсер (~250 строк Python через xml.etree)
- `metadata_utils/predefined_parser.py` — Predefined.xml
- `metadata_utils/metadata_types.py` — 44 типа объектов + BSL mapping
- `metadata_utils/metadata_repository.py` — кэш + прогрессивная загрузка
- `metadata_utils/role_parser.py` — Rights.xml роли
- `metadata_utils/subsystem_membership.py` — принадлежность подсистемам
- `data/xsd/*.json` — 27 JSON-схем XDTO
- `data/field_values.json` — справочник допустимых значений 442 свойств

## ADR-003 · Нормализатор + валидатор имён в _meta_edit
**Дата:** 2026-05-15
**Контекст:** Реальный сценарий: LLM передал `value="Город Строка(50)"` в `meta_edit add-attribute`. PowerShell-скилл принял всю строку как имя реквизита, создал XML с `<Name>Город Строка(50)</Name>`. `db_load` упал на XDTO-валидации NCName. Не критично за пара итераций, но в БД попал мусор.

**Варианты:**
- A. Полагаться на LLM (улучшить docstring)
- B. Нормализация value до PS-скилла
- C. Серверная валидация имени до PS-скилла
- D. Всё вместе (A+B+C)

**Выбор:** D — глубокая защита
**Обоснование:**
- Один уровень защиты недостаточен (LLM ошибается)
- Нормализатор покрывает 95% случаев (русские типы → английские, пробел → двоеточие)
- Валидатор имени отлавливает оставшиеся 5% (LLM передал что-то совсем странное)
- Расширенный docstring сообщает LLM правильный формат

**Покрытие тестами:**
- 23/23 кейса нормализации (Город Строка(50) → Город:String(50) и т.п.)
- 15/15 кейсов валидации (пустое / с цифры / со скобкой / > 128 символов / ...)
- 10/10 кейсов извлечения имён из value (для add-attribute / add-ts / batch / ...)

## ADR-005 · Семантический поиск через ChromaDB + USER-bge-m3
**Дата:** 2026-05-15
**Контекст:** В Agenter был только FTS5-точный поиск по справке (`platform_doc_lookup`). LLM-агент ищет «как заблокировать данные» → не находит `БлокировкаДанных` (FTS5 не понимает синонимы). Нужно semantic-расширение.

**Варианты:**
- A. Не делать (полагаться на точный поиск)
- B. Подключить чужой MCP-сервер с RAG (comol/1c_help_mcp)
- C. Свой ChromaDB-индекс поверх существующего platform_docs.db

**Выбор:** C
**Обоснование:**
- Не дублируем парсинг .hbk (берём уже распарсенную SQLite)
- Контроль над моделью embedding и качеством
- Никаких внешних зависимостей (нет Docker, нет лицензий)
- ChromaDB + sentence-transformers — open-source стек, упаковывается в PyInstaller

**Выбор модели:** `deepvk/USER-bge-m3`
- bge-m3 — SOTA multilingual, контекст 8192 токенов
- USER-* — fine-tune deepvk на русском, SOTA для русских RAG-задач
- Размер: ~1.37 ГБ (приемлемо для дистрибутива)
- Apache-2.0 лицензия

**Метрика:** cosine (через `hnsw:space: cosine` в ChromaDB), с `normalize_embeddings=True` при encode.

## ADR-006 · БСП через CommonModules, не через .hbk
**Дата:** 2026-05-15
**Контекст:** Изначально планировалось индексировать справку БСП из отдельного .hbk файла. Разведка показала: .hbk БСП в стандартной установке не существует (распространяется через ИТС-подписку отдельно).

**Варианты:**
- A. Требовать у пользователя скачать .hbk БСП с ИТС
- B. Парсить BSL-комментарии (DocString'ы) из CommonModules/*/Ext/Module.bsl
- C. Не делать БСП (фокус только на платформенной справке)

**Выбор:** B
**Обоснование:**
- DocString'ы БСП — стандартизованы (Параметры/Возвращаемое значение/Пример)
- БСП уже встроена в ERP/УТ/УНФ → у нас уже есть выгрузка в SCHEME
- Работает универсально на ЛЮБОЙ конфигурации (не только ERP):
  - Демонстрационное Приложение → индексирует свои общие модули
  - ERP → индексирует БСП + кастомные модули ERP
  - Произвольный проект → индексирует кастомные общие модули
- Самая свежая версия — синхронно с установленной БСП
- Опционально: команда «Индексировать внешнюю папку с .bsl» — для свежайшей БСП с другого источника

## ADR-007 · GPU-ускорение через PyTorch CUDA + auto-detection
**Дата:** 2026-05-15
**Контекст:** Индексация 25 509 записей на CPU занимает ~40 минут (10 doc/sec на USER-bge-m3). У пользователя RTX 3050 Laptop (4 ГБ VRAM, CUDA 12.6 driver). При переходе на GPU: ~80-150 doc/sec → 3-5 минут.

**Варианты:**
- A. Только CPU (простой PyInstaller, ~200 МБ torch)
- B. Только CUDA (быстро, но требует NVIDIA, ~2.5 ГБ torch)
- C. Auto-detection (CPU по умолчанию, переход на CUDA если установлен)

**Выбор:** C
**Обоснование:**
- `SEMANTIC_DEVICE` ENV-переменная позволяет принудительно задать
- `_detect_device()` пробует cuda → mps (Mac) → cpu
- При установке `torch+cuXXX` через `scripts/install-cuda-torch.ps1` — auto работает
- Если CUDA не установлена — graceful fallback на CPU без падений
- В дистрибутив идёт CPU-вариант по умолчанию (~200 МБ); GPU — опциональное расширение пользователя

## ADR-008 · File-lock против параллельных индексаций
**Дата:** 2026-05-15
**Контекст:** Когда мы запустили `scripts/run_full_chroma_index.py` в фоне и параллельно пользователь нажал «Построить семантический индекс» в UI → оба процесса вызвали `client.delete_collection()` → второй процесс упал с `NotFoundError: Collection [...] does not exist`.

**Варианты:**
- A. Документировать «не запускать дважды» (хрупко)
- B. File-lock (PID + timestamp)
- C. Распределённый lock через ChromaDB (overkill)

**Выбор:** B
**Обоснование:**
- File-lock прост в реализации
- Проверка живости PID — `OpenProcess` на Windows, `os.kill(pid, 0)` на Unix
- Stale-lock от мёртвого процесса автоматически снимается
- `try/finally` гарантирует снятие lock при любом завершении

**Файл lock:** `data/platform_docs_chroma.lock` с содержимым `<PID>|<timestamp>`.

## ADR-017 · Переключатель модели в UI + расход токенов в логах
**Дата:** 2026-05-15
**Контекст:** До этого момента модель была захардкожена в `orchestrator_sdk.py:653` как `claude-sonnet-4-6`. Юзер не имел возможности выбрать другую модель на конкретную задачу. Также расход API-токенов отображался только при ошибке (через `final.usage`), на успешной задаче — не показывался вообще. Это неудобно для оценки стоимости задачи и для выбора правильной модели под сложность.

Пользователь явно попросил минималистичный переключатель в composer-bar (как в Claude Code Desktop) + отображение токенов «хотя бы примерно».

**Варианты для UI переключателя:**
- A. Полноценный select с описаниями каждой модели — занимает место, шумит
- B. Segmented control (3-4 кнопки в ряд) — компактно, но не масштабируется на 6+ моделей
- C. **Текст + ▾ с popup-меню** — Claude Code Desktop делает именно так. Минимум шума, расширяется при необходимости

**Выбор:** C
**Обоснование:**
- Соответствует UX-паттерну Claude Code Desktop, юзер уже знает как это работает
- Лёгкое расширение: добавил модель в `ALLOWED_MODELS` на бэке → автоматически появляется в popup
- localStorage сохраняет выбор между сессиями
- Цена за 1M токенов отображается рядом с названием модели — даёт юзеру контекст для выбора

**Варианты для Auto-режима:**
- D. Псевдо-Auto: эвристика по ключевым словам в промпте (sophisticated/refactor → Opus, простой → Haiku, иначе Sonnet)
- E. Заглушка: Auto = Sonnet 4.6 с пометкой «TODO: routing»
- F. Настоящий routing через классификацию sub-agent'ом (+1 вызов Haiku в начале)
- G. **Пока не включать**

**Выбор Auto:** G на сейчас, F на будущее (Фаза 6 PRO-edition)
**Обоснование:**
- Юзер выбрал G после обсуждения — нет смысла усложнять без реальных данных о паттернах задач
- Псевдо-Auto (D, E) даёт ложное ощущение интеллекта без реальной пользы
- Настоящий routing требует тестирования на десятках задач — не сейчас
- Добавлено в PROGRESS.md как UX-backlog пункт «Auto-режим выбора модели»

**Варианты для отображения usage:**
- H. Только финальный итог в конце задачи
- I. **Per-turn + финальный итог** (тонкая мета-строка после каждого AssistantMessage + сводка «📊 Итого» в конце)
- J. Per-token в реальном времени (через streaming) — overkill, SDK всё равно отдаёт usage только в конце ответа модели

**Выбор:** I
**Обоснование:**
- Per-turn даёт юзеру видеть «куда уходят деньги» — какой шаг был дорогим
- Цена считается из `MODEL_PRICING` (наша табличка) ИЛИ из `ResultMessage.total_cost_usd` если SDK его посчитал (последнее точнее: учитывает фактические тарифы Anthropic)
- CSS `exec-row.usage`: opacity 0.55, font-size 10.5px, italic — не кричит в логе, видно только если присматриваешься

**Реализация:**
- `app/main.py`:
  - `ALLOWED_MODELS = {"sonnet-4-6": "claude-sonnet-4-6", "opus-4-6": "...", "haiku-4-5": "..."}`
  - `MODEL_PRICING` — цены за 1M токенов (in/out/cache_read/cache_write)
  - `_resolve_model(req.model, cfg)` валидирует и возвращает API alias
  - `TaskRequest.model: str | None` + новый endpoint `GET /models`
- `app/orchestrator_sdk.py`:
  - `run_task(model=..., model_pricing=...)`
  - `ClaudeAgentOptions(model=effective_model)` — динамическое значение
  - `_stream_messages(model_pricing=...)` — логирует per-turn usage + считает $
  - `_fmt_tokens(n)` (1.5M / 12.3k / 999) + `_calc_cost_usd(usage, pricing)`
  - Финальная «📊 Итого» строка показывается всегда (и на успехе, и на ошибке)
- `frontend/chat-screen.jsx`: компонент `ModelPicker` (popup со списком), state `models` + `selectedModel` (с localStorage hydration); `window.__selectedModel` для vanilla JS моста
- `frontend/agenter-connect.js`: `AgenterAPI.createTask(prompt, "erp", model)` — пробрасывает выбранную модель
- `frontend/styles.css`: `.model-picker-*` (минималистичный popup) + `.exec-row.usage` (мелкий серый стиль)

**Trade-off:** ~300 строк кода. Зато юзер видит сколько стоит каждая задача и может осознанно выбирать модель.

**Связано с:** ADR-011 (Sonnet 4.6 как дефолт + multi-model routing в будущем), ADR-016 (бюджет turns — теперь логично связать с бюджетом в $).

## ADR-016 · Раздельный бюджет turns: «эффективные» vs ask_user
**Дата:** 2026-05-15
**Контекст:** После добавления `ask_user` (ADR-013) встал вопрос: должен ли вызов ask_user расходовать общий лимит `max_turns`? Пользователь сформулировал: «если агент спросил у юзера и остановился, то счётчик итераций должен сбрасываться?». Технические факты:
- SDK 0.1.81 принимает `max_turns` один раз в `ClaudeAgentOptions`, динамическая правка невозможна.
- ask_user это блокирующий tool: SDK ждёт его resolution. Время ожидания юзера НЕ тратит turns (важно: юзер может думать 5 минут).
- Но **сам вызов** ask_user (assistant пишет tool_use → tool_result → assistant продолжает) — это 1 turn в SDK conversation, ровно как любой другой tool.

**Проблема:** Если юзер ответил на 70-м turn'е (при max_turns=80), у агента осталось 10 шагов на завершение. Уточнение получается «штрафным» — каждое ask_user отнимает шаг у полезной работы. Это создаёт неявное давление «лучше угадать, чем спросить», что противоречит цели tool'а.

**Варианты:**
- A. Оставить как есть (ask_user = 1 turn) — простой, но уточнение «дороже» чем должно быть
- B. Бонус +N turns после ответа юзера — идиоматично, но SDK не позволяет менять max_turns динамически
- C. Не считать ask_user за turn — самый честный, требует своего счётчика
- D. Полный сброс счётчика после ответа — самый щедрый, но риск бесконечности
- E. Просто поднять MAX_ITERATIONS до 120 — закроет 90% случаев, но не решает корневой вопрос

**Выбор:** **Гибрид B+E**
**Обоснование:**
- B+E не противоречат друг другу: E даёт запас, B-семантика делает уточнения «бесплатными» по бюджету
- C технически = B (оба ведут свой счётчик), но C говорит «0 turns на ask_user», а B — «+N turns каждое ask_user». В сумме на 5 уточнений: C даёт 120, B (с N=5×4=20) даёт 100+20=120. Эквивалентно.
- D отвергнут: без верхней крышки на ask_user можно зациклиться (агент спрашивает → отвечает → опять упирается). Мы добавляем эту крышку (5/задача), но в B-семантике она органична: «5 свободных уточнений на задачу».

**Реализация:**
- `backend/main.py`: `MAX_ITERATIONS = 120` (эффективный бюджет), `MAX_ASK_USER_PER_TASK = 5` (лимит уточнений), `SDK_HARD_CAP_TURNS = 200` (страховочный SDK-лимит)
- `app/orchestrator_sdk.py`: mutable `turn_state = {"effective": 0, "ask_user": 0, "warned_80": False}`, общий для pre и post hooks
- PostToolUse: инкрементирует `effective` для всех tools **кроме** ask_user; инкрементирует `ask_user` только для ask_user. Лог-предупреждение на 80% эффективного бюджета (один раз).
- PreToolUse: при `effective >= max_effective_turns` блокирует ВСЕ tools с сообщением `BUDGET EXHAUSTED — заверши задачу финальным текстом`. При `ask_user >= 5` блокирует только ask_user (остальное работает).
- SDK `max_turns` = `max(SDK_HARD_CAP_TURNS, max_effective + MAX_ASK_USER + 20)` — потолок намного выше нашего, чтобы наш счётчик всегда срабатывал первым.
- Системный промпт обновлён: «ask_user не тратит бюджет, но максимум 5 на задачу; при BUDGET EXHAUSTED — финальный ответ юзеру».

**Что юзер видит:**
- В начале задачи: «Бюджет turns: 120 эффективных + 5 ask_user (SDK cap 200)»
- На 80% эффективного бюджета: «⚠ Бюджет turns на 80% (96/120). Скоро останется только финальный ответ — заверши важное»
- При исчерпании: «⛔ <tool>: BUDGET EXHAUSTED» + агент завершает задачу финальным текстом

**Соответствие техническим ограничениям SDK 0.1.81:**
- SDK turn counter мы не контролируем, но он у нас завышен (200) — наш счётчик упирается первым
- Реальная остановка через PreToolUse `permissionDecision: deny` — агент получает is_error и решает что делать (как правило завершает задачу финальным сообщением)

**Trade-off:** Чуть больше кода в hooks (~60 строк). Зато прозрачно: ask_user явно не штрафует, лимиты понятные, нет магических цифр в SDK options.

**Будущая эволюция (см. PROGRESS.md AQ-2):**
Текущее решение (B+E) — компромисс для бесплатного личного использования: ограничиваем расход API-токенов через искусственный лимит turns. Если/когда появится коммерческая версия с собственной системой токенов / биллингом / BYOK (Фаза 7 PRO-edition):
- Логика меняется: юзер платит за реальные API-токены сам — нет смысла ограничивать его турнами
- Переход на **вариант D** становится естественным: после каждого ответа юзера счётчик turns сбрасывается, агент может работать сколько нужно (юзер сам останавливает кнопкой «Прервать» если задача затягивается)
- Альтернативно: модель «лимит = бюджет в токенах», который юзер сам выставляет в настройках задачи; счётчик идёт по фактическому расходу `usage.input_tokens + usage.output_tokens`, не по turns
- `MAX_ASK_USER_PER_TASK = 5` тоже можно снять — если юзер согласен на петлю, это его деньги

Триггер для рефакторинга: момент внедрения биллинга. До этого B+E остаётся.

## ADR-015 · Где жить доработке: расширение vs основная конфигурация (СТАТУС: OPEN)
**Дата открытия:** 2026-05-15
**Статус:** 🟡 OPEN — решение откладывается до накопления опыта на 2-3 реальных ТЗ

**Контекст:** Сейчас Agenter работает с одним `ext_src_path` — папкой расширения. Любая правка идёт в это расширение через CFE-механику (`cfe_borrow` / `meta_compile`). На демо-конфигурации и большинстве задач от пользователей это правильный путь: расширение = безопасно (легко откатить), не требует поддержки со стороны вендора, переживает обновления типовых.

Но это **не универсально**. Реальные сценарии, где CFE не подходит или ограниченно подходит:

| Сценарий | Что нужно | Подход CFE | Прямая конфигурация |
|---|---|---|---|
| Добавить справочник АП_ФизическиеЛица + регистры + доп. реквизиты к РасходТовара | новые объекты + заимствование | ✅ идеально | ⚠ создаст техдолг |
| Удалить реквизит из типового справочника | структурное изменение | ✗ CFE не умеет удалять | ✅ только так |
| Сменить тип значения у типового реквизита | смена типа | ✗ ограниченно (Type из CFE наследуется, но не заменяется) | ✅ |
| Глобальный рефакторинг кода вендора в общих модулях | большие изменения BSL | ⚠ можно через `&Вместо`, но громоздко | ✅ |
| Изменение прав/ролей на типовые объекты | новые роли | ⚠ CFE даёт ограниченные права | ✅ для полного контроля |
| Внедрить интеграцию через свой регистр + свои документы | сетка собственных объектов | ✅ идеально | ⚠ привязка к вендорскому циклу |
| Добавить в УЖЕ существующее чужое расширение | редактирование external CFE | — | — нужен отдельный режим |

**Открытые подвопросы:**
1. **Критерии выбора** — нужен формальный чек-лист, по которому агент и человек одинаково оценивают «куда это положить». Сейчас агент по умолчанию идёт в CFE, что часто правильно — но не всегда.
2. **Механизм управления режимом**:
   - Ручной переключатель в Sidebar (Mode: `CFE / Конфигурация / Other-Ext`)
   - Указание в промпте/командой (`/mode:config`, `/mode:ext=ВнешнееРасширение`)
   - Автоматическое уточнение через `ask_user` если из контекста неясно (после ADR-013 это технически просто)
   - Скорее всего **комбинация всех трёх**: дефолт + override промптом + ask_user на сомнениях
3. **Поддержка нескольких расширений в одной БД** — сейчас в `config.json` один `ext_src_path` и одно `extension`. Реальные 1С-базы часто имеют 3-10 расширений (типовых, кастомных, от подрядчиков). Нужен ли реестр?
4. **Безопасность правки основной конфигурации**: нет дешёвого отката (`Удалить расширение`). Гард на `db_load` для конфигурации должен требовать явного подтверждения (диалог, не просто текст в промпте).
5. **Маршрутизация tools**: `meta_compile` / `cfe_borrow` сейчас жёстко знают про CFE. При прямой работе с конфигурацией — нужны параллельные tools (`config_compile`, `config_edit`), или общие с режим-параметром?
6. **Интеграция с db_dump/db_load**: PowerShell-скилл сейчас работает с CFE через `-Extension`. Для основной конфигурации — скрипт другой (`-Mode Full -Configuration`).

**Варианты решения** (для будущего ADR):
- A. **CFE-only forever** — отказаться от прямой работы с конфигурацией, любой нестандартный сценарий → отказ агента с пояснением. Самое безопасное, но потенциально режет 20-30% задач.
- B. **Mode-switch (статичный)** — переключатель в UI на момент создания задачи; в рамках задачи режим не меняется.
- C. **Mode-aware агент** — агент сам определяет режим по характеру задачи (или спрашивает через ask_user), может смешивать (часть в CFE, часть в config) в одной сессии — но это требует жёсткой защиты от ошибок.

**Когда выбирать:** после 2-3 реальных прогонов больших ТЗ. Сейчас нет достаточно данных, чтобы понять какой режим встречается чаще и где CFE-only действительно становится тормозом.

**Связано с:** ADR-013 (ask_user — нужен для авто-уточнения режима), ADR-009 (PRO-edition: упаковка определит требования к multi-ext поддержке).

## ADR-013 · ask_user как первоклассный SDK-tool (HITL)
**Дата:** 2026-05-15
**Контекст:** Реальный прогон большого ТЗ (9 шагов, видео-промпт «АП_ФизическиеЛица + регистры + доработка РасходТовара») закончился `error_max_turns` за ~7 минут. Главные паттерны сбоя:
- LLM упёрлась в ложный PRE-FLIGHT BLOCKED (см. ADR-014) и в течение 5 turns пыталась обойти его вариантами одного meta_compile.
- При очередной непонятной ошибке (`meta_edit set` не поддерживается, `Write` для XML заблокирован) LLM продолжала гадать дальше, потому что **у неё не было способа спросить юзера**.
- Цепочка ошибок множилась, лимит ходов кончился.

Пользователь сформулировал требование явно: «нужно добавить функционал как в Desktop Claude Code — агент задаёт вопрос/уточнение, или с вариантами ответов, или просто ждёт ответ в свободной форме».

**Варианты:**
- A. Положиться только на системный промпт «при тупике останавливайся и пиши финальный ответ» — модель и так должна это делать, но не делает
- B. Тулзу `ask_user`, которая просто возвращает «спросите юзера» — без блокировки (ассимметрично: модель спросила, но ответ получит не сразу)
- C. **Блокирующий tool `ask_user`** — встаёт в `event.wait()`, UI получает WS-сообщение, юзер отвечает через REST → tool разбуживается → возвращает ответ обратно в LLM. Полный HITL (human-in-the-loop) цикл

**Выбор:** C
**Обоснование:**
- Соответствует идиоме Claude Code Desktop: «попроси у пользователя — продолжи»
- Не ломает агент-loop (max_turns не тратится, пока ждёт): SDK видит, что tool ещё работает — не считает за turn
- Чистый failure-mode: при cancel задача отменяется, tool возвращает sentinel `[CANCELLED: ...]`; при timeout (10 мин) возвращает `[TIMEOUT: ...]`. LLM умеет интерпретировать и не зацикливается
- options-массив даёт UI-radio для типовых развилок (выбор префикса, выбор справочника); без options — свободный textarea с Ctrl+Enter

**Реализация:**
- `app/main.py:_ask_user_async` — регистрирует `pending_questions[task_id]` с `asyncio.Event`, шлёт WS `ask_user`, ждёт `event.wait()` с timeout 600 сек
- `app/main.py: POST /tasks/{id}/answer` — кладёт answer + event.set()
- `app/main.py: cancel_task` — при отмене сначала разбуживает висящий ask_user с answer=None (sentinel `[CANCELLED]`), потом cancel handle
- `app/sdk_tools.py:make_agenter_tools(executor, ask_user_ctx=...)` — регистрирует @tool ask_user только если переданы task_id+callback (иначе агент не сможет спросить)
- `app/sdk_tools.py` JSON Schema: `question` (3-2000 chars) + опц. `options` (массив 2-6 строк)
- `app/orchestrator_sdk.py:run_task(on_ask_user=...)` — пробрасывает factory в make_agenter_tools на каждом запуске задачи
- `frontend/chat-screen.jsx:AskUserDialog` — overlay с blur, radio (если options) или textarea (если нет), две кнопки: «Прервать задачу» (cancel) и «Ответить»
- `frontend/agenter-connect.js` — WS handlers `ask_user` / `ask_user_resolved`
- `frontend/api.js:submitAnswer(taskId, answer, qid)` — POST на бэк
- Системный промпт обновлён: когда вызывать, когда НЕ вызывать, правило повторов

**Гарантии:**
- Один task_id ↔ один висящий вопрос (новый ask_user не задаётся, пока предыдущий не разрешился)
- qid (`secrets.token_hex(8)`) — защита от race condition «юзер ответил на старый вопрос после того как пришёл новый»
- Лог Q+A автоматически попадает в exec-row как text-block (история сохраняется в БД задачи)

**Trade-off:** +1 round-trip в задачу (когда tool используется); ~250 строк кода (backend+frontend). Это окупается за один спасённый сбой `error_max_turns`.

## ADR-014 · Динамический pre-flight кэш known_objects
**Дата:** 2026-05-15
**Контекст:** Прогон большого ТЗ показал критический баг: после успешного `meta_compile` справочника АП_ФизическиеЛица следующий `meta_compile` регистра со ссылкой `CatalogRef.АП_ФизическиеЛица` блокировался PRE-FLIGHT с «нет в основной конфигурации». Проблема: `_list_known_objects` в `tool_guards.py` корректно объединяет SCHEME + ext_src — но `orchestrator_sdk.run_task` собирал словарь **ровно один раз** до старта LLM-цикла и пробрасывал immutable снимок в hook через closure. Файл свежесозданного объекта появлялся в `ext_src/Catalogs/`, но preflight видел старый снимок.

LLM не понимая природы сбоя, повторила тот же meta_compile в 5 вариантах (batch, разные форматы типов, прямой Write XML — это упёрлось уже в GUARD по .xml). На это ушло ~8 turn'ов из 50 → главная причина `error_max_turns`.

**Варианты:**
- A. Пересобирать `known_objects` перед каждым PreToolUse hook — корректно, но лишний scan диска (тысячи файлов) на каждый tool call
- B. Mutable container `known_objects_ref = {"data": ...}`, hook читает `ref["data"]`; PostToolUse hook пересобирает после успешных «структурных» tools (meta_compile / cfe_borrow / cfe_patch_method)
- C. Полностью убрать preflight для ext_src — слишком сильное упрощение, мы потеряем защиту от опечаток в именах основной конфигурации

**Выбор:** B
**Обоснование:**
- Сканирование выполняется ровно тогда, когда **может** появиться новый объект (после успешного meta_compile/cfe_borrow). Не на каждый Read/Grep
- Mutable container — минимальное изменение API: `check_tool_call(known_objects=...)` принимает то же значение, просто из `ref["data"]`
- В UI логируется «Pre-flight кэш обновлён: 37 объектов (+1 после meta_compile)» — юзер видит, что система реагирует на изменения
- Защита от опечаток в SCHEME-ссылках сохраняется

**Реализация:**
- `_make_pre_tool_use_hook(known_objects_ref: dict)` — читает `known_objects_ref["data"]` на каждом вызове
- `_make_post_tool_use_hook(known_objects_ref, client_cfg)` — после успешных tools из `_TOOLS_THAT_AFFECT_KNOWN_OBJECTS` вызывает `_list_known_objects()` заново и обновляет `ref["data"]`
- Учёт is_error из tool_response (раньше PostToolUse считал любой вызов успешным — теперь различаем)
- При delta != 0 — лог в UI

**Связано с:** ADR-013 (ask_user). Эти два фикса вместе закрывают главные паттерны сбоев из лога 2026-05-15.

## ADR-012 · Структурированный JSON API вместо DSL-строк
**Дата:** 2026-05-15
**Контекст:** Реальный сценарий: Claude передал `value="Город Строка(50)"` в `meta_edit`. Это формально валидно с т.з. типа параметра (string), но семантически — Claude не разделил имя и тип на компоненты по нашему DSL-формату `Name:Type[|flags]`. 3-слойная защита (нормализатор + валидатор + расширенный docstring) — это **workaround**, не системное решение. Пользователь справедливо заметил: «не будем же мы каждую возможную ошибку обрабатывать регулярками».

**Корневая причина:**
- DSL `"Имя:Тип(длина)|флаги"` — это **наша локальная придумка**, нигде не задокументированная в обучающем корпусе Claude
- ML-модель мимикрирует под стиль пользователя (если в промпте «Строка(50)» — Claude передаёт «Строка(50)»)
- Каждый новый вариант форматирования = новая регулярка → техдолг

**Открытие:** скилл `meta-edit.ps1` уже поддерживает `-DefinitionFile <json>` режим со структурированным JSON. То есть DSL — это только формат **inline**-режима. JSON режим использовался слабо.

**Варианты:**
- A. Развивать DSL-нормализатор (всё больше regex'ов)
- B. Сделать 8+ узких структурированных tools (`meta_add_attribute`, `meta_add_ts`, и т.д.)
- C. Один structured tool с богатой JSON Schema, передающий JSON напрямую в `-DefinitionFile`

**Выбор:** C
**Обоснование:**
- Минимум кода: один tool, одна JSON schema, никаких dsl-конвертеров
- LLM **физически не может** передать невалидное имя — JSON schema с `pattern` валидируется в Anthropic API ДО запроса в наш код
- `enum` для перечислимых типов (indexing/fillChecking) — LLM не может ошибиться
- PS-скилл получает чистый JSON, не нужно ничего парсить
- Старые защиты (3-слойная нормализация) остаются как **defense in depth** для редких случаев

**Реализация:**
- `app/sdk_tools.py:meta_edit` — JSON schema с pattern для имён, enum для типов индексации
- `desktop/main.py:_meta_edit` — принимает `dict` definition, валидирует имена через `_validate_definition_names`, пишет temp.json, вызывает `meta-edit.ps1 -DefinitionFile`
- `_validate_definition_names` — рекурсивный обход definition, валидация всех name полей через `_validate_meta_name`
- DSL-обёртка (`_normalize_meta_edit_value`, `_extract_names_from_value`, `_normalize_ts_shorthand`) — **больше не вызывается** из основного пути, но остаётся в файле как escape hatch (если когда-то понадобится inline-режим скилла)

**Покрытие тестами:**
- 17/17 кейсов валидации definition имён (add/modify/remove + все типы children)
- UTF-8 JSON запись (без \uXXXX escape)

**Trade-off:** docstring tool стал больше (~200 строк JSON schema), но это документация которая работает на LLM, не на нас.

## ADR-010 · BATCH_SIZE=8 + memory cleanup для RTX 3050 (4 ГБ VRAM)
**Дата:** 2026-05-15
**Контекст:** При полной индексации 25k записей на RTX 3050 Laptop (4 ГБ VRAM) с BATCH_SIZE=32 процесс встал на 15,500/25,509 (60%) и через час не сдвинулся. CPU 100%, GPU 100% util, но count не растёт. Диагноз: RAM забита (12.8/15.7 GB) + PageFile 29.8 GB → активный своп → каждая операция blokируется I/O.

**Корневая причина:**
- USER-bge-m3 = 1.37 ГБ в VRAM
- BATCH_SIZE=32 buffer ≈ 1.5 ГБ доп. VRAM
- Итого VRAM 94% — фрагментация и активный allocator cleanup
- HNSW-индекс на 15.5k векторах в RAM ~500 МБ
- Encode tensors + Python overhead ещё ~2-3 ГБ RAM
- Накопление → своп → катастрофа скорости

**Варианты:**
- A. Уменьшить BATCH_SIZE с 32 до 8
- B. Использовать модель помельче (multilingual-e5-small 440 МБ)
- C. Откатиться на CPU (медленнее но без swap)

**Выбор:** A + memory cleanup
**Обоснование:**
- BATCH_SIZE=8 для bge-m3 на RTX 3050 Laptop оставляет ~2 ГБ свободного VRAM
- CHROMA_INSERT_BATCH=200 (вместо 500) — реже накапливается HNSW в RAM
- После каждого batch: `gc.collect()` + `torch.cuda.empty_cache()` — освобождает буферы
- На GPU 8+ ГБ можно поднять обратно до 32-64 — будет быстрее

**Trade-off:** скорость на batch_size=8 ~80-120 doc/sec vs 280 при batch=32. Но 0 doc/sec из-за свопа — хуже любого варианта.
