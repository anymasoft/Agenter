# Аудит Agenter и роадмап — 2026-05-15

> Полный аудит после 5 неудачных запусков Промпт.txt подряд. Цель — найти все архитектурные дыры, понять почему агент проигрывает Cursor/Antigravity на больших задачах, и составить план, как это починить.

## TL;DR

**Главная находка.** В проекте лежит 66 готовых скиллов 1С и 8 готовых MCP-серверов. Агент использует **7 скиллов и 1 сервер** — это меньше 12% доступного. На больших задачах он отрабатывает в Bash/Edit/Grep там, где должен был дёрнуть готовый инструмент — поэтому уходит в 5-минутные циклы и упирается в бюджет turns.

**Корневая причина пяти провалов Промпт.txt** — не модель и не SDK, а **бедный набор tools у агента**:

| Что у нас в `.claude/skills/` | 66 скиллов |
|---|---|
| Что подключено к агенту | 7 (cfe_validate, cfe_borrow, cfe_patch_method, meta_compile, meta_edit, db_dump, db_load) |
| Что лежит в `MCP серверы/` | 8 готовых серверов |
| Что подключено к агенту | 1 (BSL Atlas) |
| Эталонные паттерны из `repo/` | 7 источников (cc, kiro, openspec, memory bank, spek-kit, taskmaster, mcp) |
| Сколько приёмов оттуда заимствовано | 0 |

### 1.2. Топ-5 корневых причин

**Причина 1. Бюджет 50 turns против ~80 нужных** *(4 из 5 запусков)*
- 9-шаговый Промпт.txt физически не помещается. Шаги 1-4 (метаданные + borrow + регистр) — это 35-45 turns; на форму + проведение + отчёт остаётся 5-15.
- Цитата: `BUDGET EXHAUSTED: эффективный лимит turns (50) исчерпан`.
- Половина turns уходит на ручную работу с XML формы, которую решает один вызов `form-edit`.

**Причина 2. Form.xml — главный обрыв** *(4 из 5)*
- На диске лежат `form-add`, `form-edit`, `form-info`, `form-validate`, `form-compile`, `form-patterns`. Агент их **не видит** (не подключены через MCP).
- Правит Form.xml через `Edit` и `Bash python` с регексами. 9 кб XML → 5-10 turns на одну правку.
- Падает на `Неверный путь к данным: "Объект.Товары.Количество"` (`567ad4ec`) — `form-validate` поймал бы локально, без `db_load`.

**Причина 3. PRE-FLIGHT loop на типах расширения** *(`46689dc9`: 15 turns подряд)*
- `meta_compile InformationRegister` со ссылкой на `CatalogRef.АП_ФизическиеЛица` блокируется, потому что справочник ещё не существует.
- Workaround агент находит сам (заглушка `String(36)` → потом `meta_edit`), но тратит на это половину бюджета.
- **Архитектурно неправильно**: PRE-FLIGHT должен либо понимать «объект будет создан в этой же задаче» (batch-режим с topological sort), либо предлагать порядок.

**Причина 4. «Слепой» поиск имён через Bash+os.walk** *(`e163672c`: 6 turns на поиск имени формы)*
- Имена папок SCHEME в cp1251, питон-скрипты выводят `????`. Агент 5 раз меняет команду.
- В итоге решает через `bsl_search_function("ПриОткрытии")` — нашёл с первого раза. Хороший инструмент был доступен с начала, но system_prompt не учит «сначала bsl_search_function, потом — никогда не Bash+os.walk по SCHEME».

**Причина 5. «Silent partial»** *(`950d7d4b`)*
- 11 минут работы, `status=done`, `db_load_count=0`, `phases_committed=0`. Агент **ни разу не загрузил в БД**, UI показал «готово».
- Это та же дыра, под которую был ADR-018. Для legacy-задач (без `session_id`) `final_state` остаётся `None` и UI снова врёт. Полузакрыто.

### 1.3. Мелкие, но красноречивые находки

- **Опечатка в пути MEMORY.md**: агент в начале каждой задачи лезет в `C:\Users\User\.claude\projects\C--BUFFER-ERP-ext-src\memory\MEMORY.md` — папка с лишним `-ext-src`. -1 turn на старте каждой задачи.
- **Нет watchdog на тишину SDK** — то самое зависание из последнего инцидента. Anthropic SSE замолчал → ждём вечно.
- **Жёсткое зашитое имя проекта `erp`** — несмотря на UI «МоеРасширение / ДЕМО 8.3», на бэке project_id всегда `erp`.

## 3. Архитектура решения

### 3.1. Главный принцип

**Агент не должен изобретать. Агент должен выбирать из меню.**

Меню — это полный каталог 66 скиллов + 9 MCP-серверов + 8 встроенных Claude-tools. Агент видит каталог одной строкой на инструмент, выбирает по семантике задачи. Если в меню нет — только тогда Edit/Bash.

### 3.2. Три слоя инструментов

#### Слой A — Hot tools (всегда подключены, 15 шт)

Покрывают 80% операций в типичной 1С-задаче. Прописываются прямо в agenter MCP сервере как первоклассные tools.

| Tool | Скилл | Что делает |
|---|---|---|
| `db_dump` | db-dump-xml | Выгрузка БД → ext_src |
| `db_load` | db-load-xml | Загрузка ext_src → БД |
| `db_update` | db-update | UpdateDBCfg (динамическое обновление) |
| `db_run` | db-run | Открыть 1С Предприятие |
| `cfe_validate` | cfe-validate | Полная валидация расширения |
| `cfe_borrow` | cfe-borrow | Заимствовать типовой объект |
| `cfe_patch_method` | cfe-patch-method | Перехватчик метода |
| `meta_compile` | meta-compile | Создать объект из JSON |
| `meta_edit` | meta-edit | Точечно изменить объект |
| `meta_info` | meta-info | Структура объекта (реквизиты, ТЧ) |
| `meta_validate` | meta-validate | Валидация одного объекта |
| `form_info` | form-info | Структура формы (элементы, реквизиты) |
| `form_validate` | form-validate | Валидация Form.xml |
| `form_edit` | form-edit | Добавить элемент/реквизит в форму |
| `form_compile` | form-compile | Сгенерировать Form.xml из JSON |

#### Слой B — Cold skills (через registry-MCP, 51 шт)

Реже используемые скиллы доступны через 2 мета-tool'a:

```
skill_search(query: str, category?: str) → [{name, summary, category, args_schema}]
   # Ищет по индексу .claude/skills/*/SKILL.md (frontmatter + first lines)
   # Возвращает топ-5 совпадений с описанием

skill_run(name: str, args: dict) → {stdout, exit_code, duration_sec}
   # Запускает .claude/skills/<name>/scripts/<name>.py (или .ps1 fallback)
   # Аргументы передаются как CLI-параметры
```

**Что попадает в «холодный» слой:**

- `mxl-*` (4): только когда нужна печать
- `skd-*` (4): только когда отчёты
- `role-*` (3): когда права
- `subsystem-*` (4): команды/подсистемы
- `epf-*` (6), `erf-*` (4): внешние обработки/отчёты
- `cf-*` (4): полная конфигурация (вне scope расширения)
- `template-*` (2), `interface-*` (2), `help-*` (1), `img-grid` (1), `web-*` (5): редкие
- `cfe-diff`, `cfe-init`: служебные

Итого ~51 скилл в холодном слое.

#### Слой C — External MCP servers

Отдельные процессы, подключаются через `.mcp.json` или config агента.

| Сервер | Приоритет | Зачем именно сейчас |
|---|---|---|
| FormsServer | **П1** | Прямо решает Form.xml-обрыв |
| SyntaxCheckServer | **П1** | Ловит BSL до db_load → экономит turns |
| HelpSearchServer | П1 | Документация платформы вашей версии |
| Graph_metadata_search | П1 | Альтернатива BSL Atlas с графовым поиском |
| SSLSearchServer | П2 | БСП-справка (когда нужно интегрироваться с типовыми) |
| TemplatesSearchServer | П2 | Шаблоны кода (повышает качество BSL) |
| CodeMetadataSearchServer | П3 | Пересекается с BSL Atlas, низкая ценность |
| 1CCodeChecker | П3 | Логическая проверка (когда BSL стабилизируется) |

### 3.3. Изменения в SYSTEM_PROMPT

Новые «правила большого пальца», которых сейчас нет:

- **R12 (Form.xml)**: правки Form.xml — только через `form_edit` / `form_compile` / `FormsServer`. Запрещено: `Edit`/`Bash regex` по `Form.xml`.
- **R13 (BSL финал)**: после правки `.bsl` — обязательно `mcp__syntax_check__check` до `cfe_validate`.
- **R14 (поиск имён)**: имена объектов 1С — только через `bsl_*` или `metadatasearch`. Запрещено: `Bash python os.walk SCHEME` (cp1251 → крокозябры).
- **R15 (топология объектов)**: если в одной задаче создаются объекты A и B, и B ссылается на A — `meta_compile(A)` строго до `meta_compile(B)`.
- **R16 (skill_search)**: перед использованием Edit/Bash для нестандартной операции — сначала `skill_search` с запросом по существу задачи.

### 3.4. Изменения в PRE-FLIGHT

- Принимать `meta_compile` с массивом объектов как **batch с topological sort**. Если объект A в batch'е объявлен раньше B и B на него ссылается — пропустить, не блокировать.
- После каждого успешного `meta_compile` / `cfe_borrow` — **обновлять кэш known_objects инкрементально**, без перечитывания SCHEME.

### 3.5. Изменения в orchestrator

- **Watchdog на тишину SDK**: 5 минут без событий → авто-cancel с понятной ошибкой («Anthropic перестал отвечать»). Закрывает зависание из последнего инцидента.
- **Адаптивный бюджет turns**: 50 для TodoWrite ≤ 3 пунктов, 80 для 4-6, 120 для 7+. Не давать обрезать большие задачи.
- **Честный final_state для legacy**: для задач без `session_id` рендерить «Завершено (статус не отслеживался)», не «Готово».
- **Путь MEMORY.md**: убрать суффикс `-ext-src` из захардкоженного пути.

### S2.7 TodoWrite-attestation финального статуса (hotfix-2 2026-05-16)

**Проблема:** правило «status=done + db_load>0 → applied» (S2.4) сработало неуниверсально для больших задач: при BUDGET EXHAUSTED после 2 фаз из 9 SDK мог завершиться `subtype=success`, и задача показывалась как `applied`, хотя 7 пунктов остались несделанными.

**Принципиальный фикс** — без привязки к типу задачи:

- Новая колонка `tasks.incomplete_todos INTEGER` (NULL если TodoWrite в задаче ни разу не вызывался — отличает «не использовался» от «всё completed»)
- Новая колонка `tasks.todos_total INTEGER` (диагностика)
- В `orchestrator_sdk.py:_make_post_tool_use_hook` после каждого успешного `TodoWrite` парсятся `todos[].status` → считаются `pending`/`in_progress` → callback `on_todos_update(task_id, incomplete, total)` пишет в БД
- В `_compute_final_state` добавлен ПЕРВЫЙ-ПРИОРИТЕТНЫЙ кейс:
  - `incomplete_todos > 0` → `partial` (если db_load>0) / `staged` (если db_load=0) / `failed` (если status=error и db_load=0)
- В SYSTEM_PROMPT добавлено правило **R18 «Ведение TodoWrite»** — агент должен поддерживать статусы актуальными (in_progress → completed по мере выполнения), иначе сигнал станет ложным
- Колонки добавлены через ALTER TABLE миграцию в `_init_db()` — backward-compat со старыми БД

**Проверено на 10 универсальных кейсах:**

| Сценарий | Ожидание | Результат |
|---|---|---|
| Простая задача без TodoWrite (incomplete=NULL) | applied | ✓ applied |
| TodoWrite=[все completed], 1 db_load | applied | ✓ applied |
| L4 TodoWrite: 7 completed, 2 pending, 5 db_load | partial | ✓ partial |
| L4 BUDGET hit, 5 in_progress, 0 db_load | staged | ✓ staged |
| Прервано посередине, 3 pending, 1 db_load | partial | ✓ partial |
| Прервано, есть todos, но 0 db_load | failed | ✓ failed |
| Прервано без TodoWrite, без db_load | failed | ✓ failed |
| Legacy без session_id | legacy | ✓ legacy |
| Read-only без TodoWrite | staged | ✓ staged |
| TodoWrite=[все completed], 0 db_load | staged | ✓ staged |

Никакого hardcode под тип задачи. Полностью основано на универсальном сигнале — TodoWrite-самоотчёт агента.

В `run_task` после построения hooks инициализируется `stage_state_ref = {"current": _load_current_stage_from_db(task_id)}`. Mutable container — обновляется в post-hook при plan_task / при успехе expected_tool.

Helper'ы в orchestrator_sdk.py:
- `_load_current_stage_from_db(task_id)` — берёт активную/первую pending из БД
- `_advance_stage_in_db(task_id, completed_index)` — completed → next pending → in_progress

### S4.5 Pre-tool guard: stage-dispatch

В `tool_guards.py:check_tool_call` добавлены параметры `stage_dispatch_required, current_stage`. Самый ПЕРВЫЙ guard в цепочке:
- Если `stage_dispatch_required` и нет активной стадии → разрешены только plan_task, read-only, ask_user.
- Если есть активная стадия → разрешён expected_tool + ALWAYS_ALLOWED.
- Все остальное → BLOCK с сообщением «На стадии X ожидался tool Y, ты пытаешься вызвать Z. Либо смени план (plan_task), либо вызови правильный tool, либо ask_user».

### S4.6 Post-tool auto-advance

В `_make_post_tool_use_hook`: после успешного `expected_tool` стадия помечается completed, активируется следующая pending. Лог в UI: `✓ Стадия #N [kind] завершена, следующая: #N+1 [kind]`.

При успешном `plan_task` (или re-plan) — перезагрузка current_stage из БД.

### S4.7 main.py: stage-dispatch активен для L2+

В `_run_task_wrapper`: `task_cfg["_stage_dispatch_required"] = (task_level != "L1")`. Лог в UI: `🎯 Stage-dispatch активен`.

L1 — простые задачи (1-2 шага), план не нужен. L2+ — обязательно через plan_task.

### S4.7b SYSTEM_PROMPT R21 «Дисциплина диспетчеризации»

Новый блок в `backend/main.py:SYSTEM_PROMPT`:
- Объяснение потока работы (plan_task → стадии → expected_tool)
- Что разрешено на любой стадии

### S2.10 Guard на Edit XML структурных типов + anti-hallucination (hotfix-5 2026-05-16)

**Провал из лога:** на простейшей задаче «Включи справочник АП_Острова в подсистему Финансы» агент потратил 22 turns на ручную правку `Subsystems/Финансы.xml` через `Edit`, получил XML-ошибку, выдумал что «расширения 1С не позволяют добавлять Content в заимствованную подсистему», и в финале **удалил подсистему из расширения** (модифицировал Configuration.xml). Скилл `subsystem-edit` с операцией `add-content` решал задачу в 1 turn — но агент не использовал.

**Корневые причины** (универсальные, не привязаны к этой задаче):

1. **Guard на XML работал только для Write**, не для Edit. Агент обошёл его через Edit XML (R4 пропускал).
2. **R16 (skill_search перед нестандартной операцией) — мягкое правило** в системном промпте, агент его игнорирует.
3. **Нет hot tool для subsystem операций** — частая операция, должна быть первоклассной как form_* / meta_*.
4. **Нет правила против галлюцинаций ограничений платформы** — агент выдумывает «невозможно» вместо обращения к ask_user или skill_search.

**Принципиальные фиксы — без подгонок под подсистемы:**

A) **Guard XML расширен на Edit** (`tool_guards.py:check_tool_call`):
   - Блокирует Edit/Write на `*.xml` под `ext_src/`
   - Подсказка определяется маршрутом пути (helper `_xml_path_hint`):
     - `Subsystems/*.xml` → «skill_run('subsystem-edit', {operation:'add-content', value:'Type.Name'})»
     - `Forms/*/Ext/Form.xml` → «form_edit / form_compile»
     - `Roles/*/Rights.xml` → «skill_search('роль')»
     - `Configuration.xml` → «НИКОГДА вручную»
     - `Catalogs/*/*.xml`, `Documents/*/*.xml` и т.п. → «meta_edit»
     - Templates, MXL, СКД, CommandInterface — свои подсказки
   - Fallback: «skill_search('<что нужно>')» если тип не распознан

B) **`_FORBIDDEN_WRITE_PATH_FRAGMENTS` уточнён**: убрал глобальный блок `/agenter/data/` (это рабочая папка для MEMORY.md и proposals), оставил только `/agenter/data/erp/snapshots/` (zip'ы — управляются snapshots.py).

C) **Hot tool `subsystem_edit`** в sdk_tools.py — обёртка над skill subsystem-edit. Покрывает: add-content, remove-content, add-child, remove-child, set-property. Включён в AGENTER_TOOL_NAMES, добавлен в `_HOT_SKILLS` чтобы не дублировался в skill_search.

D) **R20 ANTI-ГАЛЛЮЦИНАЦИЯ ОГРАНИЧЕНИЙ ПЛАТФОРМЫ** в SYSTEM_PROMPT:
   - «Если ты собираешься сказать "расширения 1С не позволяют..." — ОСТАНОВИСЬ. С вероятностью 95% ты выбрал неправильный инструмент»
   - Перечислено что РАСШИРЕНИЯ позволяют: add-content к Adopted subsystem, реквизиты к заимствованным объектам, перехваты, формы-extension, и т.д.
   - Жёсткое правило: «3+ turn'а на безуспешные обходы → переключайся на ask_user/skill_search. НЕ удаляй заимствованные объекты "потому что не получилось" — это потеря работы пользователя»

### S2.9 Windows cmdline-limit hotfix — `--append-system-prompt-file` (hotfix-4 2026-05-16)

**Проблема:** при запуске L4-задачи с `interactive_mode=true` (например Промпт.txt) Agenter падал с ошибкой `Claude Code not found at: ...\claude.exe`, хотя файл claude.exe (227 МБ) был на месте и запускался. Простые задачи проходили.

**Корневая причина** — Windows `CreateProcess()` API имеет лимит длины командной строки **32,767 символов**. SDK Python (`subprocess_cli.py:236-238`) передаёт `--append-system-prompt <huge string>` напрямую через argv. С учётом наших добавок (R12-R19, L4 spec-first, sub-agent, MEMORY.md inline) append-блок достигает 14K символов, базовый SYSTEM_PROMPT 21K → итого 35K. Windows API возвращает `FileNotFoundError`, SDK перехватывает и преобразует в обманчивое сообщение «Claude Code not found».

**Замер на реальной L4-задаче с interactive_mode:**

```
BASE SYSTEM_PROMPT:        21 182 chars
addendum (наш sdk_addendum): 13 763 chars
ИТОГО system_prompt:       34 945 chars
Windows cmdline limit:     32 767 chars
Превышение:                 2 178 chars
```

Простой Банки.Цвет проходил, потому что L1/L2 + без interactive_mode → addendum всего ~3K → влезали в лимит.

**Решение** — monkey-patch SDK без модификации .venv:

`agenter/app/orchestrator_sdk.py:_patch_sdk_for_long_system_prompt()` оборачивает `SubprocessCLITransport._build_command`. Если значение `--append-system-prompt` или `--system-prompt` превышает порог 4000 символов — пишет во временный файл (`%TEMP%\agenter_sp_*.txt`) и заменяет на flag `-file`-вариант. claude.exe этот флаг поддерживает нативно (`--append-system-prompt-file <path>` упомянут в `claude.exe --help`).

Патч применяется один раз при импорте модуля. Идемпотентный.

**Проверено на реальном cmdline:**

| | До патча | После патча |
|---|---:|---:|
| Длина cmdline | 14 211 chars | **272 chars** |
| Превышение лимита | да (2,2K сверху) | нет |
| Файл создан | — | `%TEMP%\agenter_sp_xxx.txt` |
| Содержимое сходится | — | ✓ полное |

Принципиально универсально: работает для любых задач — простых, средних, больших. Когда промпт умещается в лимит — патч не вмешивается (порог 4K). Когда большой — прозрачно выносит в файл. Никакого hardcode под тип задачи.

### S2.8 Интерактивный режим из промпта (hotfix-3 2026-05-16)

**Проблема:** пользователь в промптах писал «если возникают вопросы — спроси у меня», но агент почти никогда не пользовался `ask_user` — описание инструмента содержит «НЕ используй для тривиальных вопросов», и эта установка перебивала инструкцию из промпта.

**Принципиальный фикс** — универсальный, без привязки к типу задачи:

- В `agenter/app/main.py:_detect_interactive_intent(prompt)` — детект явной речевой интенции по 14 универсальным паттернам RU/EN
- В `_run_task_wrapper` устанавливается `task_cfg["interactive_mode"]`
- В `orchestrator_sdk.py:_build_client_context` при `interactive_mode=true` добавляется блок **R19 «ИНТЕРАКТИВНЫЙ РЕЖИМ»** который ПЕРЕОПРЕДЕЛЯЕТ дефолтную сдержанность ask_user именно для этой задачи
- Лог в UI: `💬 Интерактивный режим — в промпте есть просьба спрашивать`

**Триггеры (позитивные):**
- RU: спроси, спросите, спрашивай, уточни, уточняй, сообщи, «если есть/возникнут вопросы», «если что-то непонятно», «не угадывай», «не выдумывай», «не додумывай»
- EN: ask me/user/first, clarify, "if you have questions", "when in doubt", "don't guess"

**Anti-патерны (блокируют режим целиком):**
- RU: «не спрашивай», «не уточняй», «не отвлекай», «разберись сам»
- EN: "don't ask/clarify/interrupt/bother", "just do it"

**Локальное отрицание** перед позитивным триггером (lookback 8 символов) — пропускает этот матч (например «не спроси» не должно срабатывать как «спроси»).

**Проверено на 22 кейсах:**
- 10 положительных, включая дословную фразу пользователя «Если возникают проблемы или вопросы, то остановись и спроси у меня уточнение» → ✓ True
- 12 отрицательных, включая хитрые с отрицанием: «Не спрашивай меня по мелочам», «Don't ask me trivial questions», «не спрашивай при сомнениях», «Just do it» → все ✓ False

**Принцип честный:**
- Не упоминал в промпте → дефолтное поведение (агент сдержан)
- Явно попросил → агент спрашивает на любой неоднозначности

Никаких прошитых имён задач, объектов или сценариев. Только реакция на универсальный речевой паттерн.
