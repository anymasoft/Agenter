## Что за проект — короткий контекст

**Agenter** — локальное десктопное приложение для разработки расширений
к платформе 1С:Предприятие через Claude Agent SDK. Архитектура:

- Один Python-процесс (`app/main.py`)
- FastAPI на `localhost:8080` + PyWebView-окно с веб-UI
- Claude Agent SDK как LLM-цикл (Sonnet/Haiku)
- 67 PowerShell-скиллов в `.claude/skills/` для модификации XML-метаданных 1С
- Sprint 4-архитектура: детерминистическая диспетчеризация стадий
  (`STAGE_KIND` enum → `STAGE_KIND_TO_TOOL` mapping → guard'ы на каждой стадии)
- BSL Atlas (SQLite-индекс конфигурации) — внешняя зависимость для поиска

Разработчик: **один человек, sole founder**. Целевой пользователь
сейчас — он же (внутреннее использование для своего ERP-проекта на 1С).
Декларируемая цель — превратить в продаваемый продукт.

Стек: Python 3.11, PowerShell 5.1, 1С 8.3.27, Anthropic API.

Подробности — в приложенных материалах.

## Что произошло в этой сессии (контекст для аудита)

Это сессия из ~24 часов работы. Ключевые сюжеты:

1. **Прод-инцидент №1.** Тест-задача «Создай справочник Посуда → Помести
   в подсистему Финансы» — первая часть прошла, вторая упала на
   `subsystem-edit` (отсутствовал `<Content>` в заимствованной подсистеме).
   Пользователь выразил резкое раздражение: «опять не использует скиллы,
   не понимает простейшую задачу».
2. **Признание методологической проблемы.** Ассистент честно ответил
   что подход «правим по одному багу» масштабируется в годы whack-a-mole.
   Пользователь согласился, дал мандат на принципиальную смену подхода.
3. **Test-first pivot.** Построена тестовая инфраструктура с нуля:
   - `tests/audit/skill_audit.py` — auto-парсер 67 SKILL.md + PS1
   - `tests/audit/scaffold_tests.py` — генератор тест-заготовок
   - `tests/harness/skill_runner.py` — обёртка над PowerShell
   - `tests/harness/golden.py`, `factories.py` — golden-files, фабрики фикстур
   - 226 тестов в 67 файлах, 65 passed, 161 skipped (с reason'ами)
4. **Закрытие двух кластеров багов** по методологии test-red → fix → test-green:
   - `subsystem-edit.ps1`: 4 симметричных бага про missing `<Content>` / `<ChildObjects>`
   - `cfe-borrow.ps1`: missing `<ChildObjects/>` для типа Subsystem
5. **Прод-инцидент №2** (после фикса №1): db_load упёрся в новый слой —
   `cfe-borrow` не пишет `<ChildObjects/>` для заимствованной подсистемы.
   Зафиксирован параметризованным тестом, исправлен.
6. **Архитектурный инвариант SYNC-FIRST** в планировщике: любой план с
   модифицирующими стадиями обязан начинаться с `sync-from-db`.
   Защищает от orphan-файлов от прошлой неудачной задачи.
   33 теста, все зелёные.

Пользователь критиковал ассистента многократно:
- За использование устаревшей терминологии («Desktop assistant» когда
  архитектура давно сменилась)
- За батники в UTF-8 (cmd.exe в CP866 — глобальная Russian Windows issue)
- За кажущееся «опять ничего не работает», когда на самом деле это был
  следующий слой багов, не остановка прогресса

Все исправления применены и зафиксированы регрессионными тестами.

## Что оценить — обязательные оси

Ниже — рамки, в которых нужен анализ. Если внутри какой-то оси у тебя
нет данных — так и напиши, не выдумывай.

### A. Продуктовая идея

A1. **Реальность проблемы.** «Разработка расширений 1С через AI-агента» —
это настоящая боль рынка, или enthusiast-нишa автора?
A2. **Конкурентная среда.** Что уже есть на рынке (cc-1c-skills, comol-*,
1С:Code Assistant, GitHub Copilot для 1С)? В чём Agenter лучше и хуже?
A3. **Moat и защита.** Что мешает кому-то скопировать подход? 67 скиллов —
это форк из cc-1c-skills (Николай Широков, GitHub), не оригинальный код.
В чём собственная ценность Agenter?
A4. **TAM/SAM/SOM.** Реалистичная оценка рынка 1С-разработчиков,
готовых платить за такой инструмент.
A5. **Sole-founder риск.** Один человек строит и продукт, и инструмент,
и пишет свой ERP в нём. Где это разваливается?

### B. Техническая архитектура

B1. **Монопроцесс vs декомпозиция.** PyWebView + FastAPI + SDK + 67 PS-скиллов
в одном Python-процессе. Что выиграно, что потеряно по сравнению с
client-server или модульной архитектурой?
B2. **PowerShell как слой модификации.** 67 PS1 скриптов с regex-парсингом
XML — насколько это устойчиво? Какие классы багов это будет рождать
бесконечно? Альтернативы (Python с lxml, .NET DOM, etc.)?
B3. **Stage-dispatch архитектура.** `STAGE_KIND` enum + `STAGE_KIND_TO_TOOL`
mapping + guard'ы на каждой стадии. Это правильный паттерн или
over-engineering? Не ограничивает ли он LLM сверх необходимого?
B4. **Зависимости.** BSL Atlas (внешний open-source проект), `cc-1c-skills`
fork, Anthropic API. Какой из них самая большая угроза проекту?
B5. **XML invariants.** Прод-инциденты показали, что PS-скиллы делали
несимметричные предположения о структуре XML. Сколько ещё таких
ловушек скрыто в 67 скриллах? Какой % реально безопасен?

### C. Методология разработки

C1. **Pivot к test-first.** Произошёл после прод-инцидента. Это правильный
момент? Late? Должен был быть с первого дня? Что значит «правильный
момент для test-first» в стартапе одного человека?
C2. **Параметризованные статические тесты vs behavioral.** Из 226 тестов
многие — static parse PS1 (например, проверка что `Subsystem` в списке).
Это разумно или это симптом ленивого тестирования?
C3. **Coverage matrix.** 67/67 файлов с тестами, но 161/226 — skip
с пометками «implement happy path». Это честное состояние или
самообман через «100% файлов покрыто»?
C4. **End-to-end тесты отсутствуют.** Что это значит на практике?
Когда они станут критичны?
C5. **Прод-bug → regression test.** Применяется ли последовательно?
Какие классы багов всё равно проскочут?

### D. Процесс и коммуникация

D1. **Стиль ассистента (Claude в этой сессии).** Слишком многословно?
Слишком оборонительно? Где конкретно ассистент был сикофантичен или
наоборот излишне самоуверен? Назови примеры из материалов.
D2. **Реакция на критику.** Пользователь несколько раз был резок
(«опять ничего не понимает»). Как ассистент реагировал — правильно?
D3. **Эскалация и принятие решений.** Где ассистент должен был
остановиться и спросить, а где наоборот — действовать без запроса?
Был ли баланс правильный?
D4. **Признание ошибок.** Ассистент в этой сессии признал ошибки в
терминологии (Desktop assistant), в кодировке (.bat в UTF-8), в выборе
архитектурного варианта (simple vs smart sync). Это здоровый паттерн
или избыточный self-flagellation?
D5. **Пользователь как менеджер.** Какие у пользователя слабости как
руководителя проекта (по сообщениям видно)? Что бы ты ему сказал,
если бы он попросил совет «как мне самому работать лучше»?

### E. Слепые пятна

E1. **Что НЕ обсуждалось** в этой сессии, но должно было? (Безопасность,
лицензирование, биллинг Anthropic API, отказоустойчивость, telemetry,
обновления, миграции БД agenter.db, …)
E2. **Тех-долг, не названный** ни одной стороной. Что в коде явно
рассыпается? Что устареет первым?
E3. **Bus factor.** Что произойдёт, если разработчик пропадёт на месяц?
Что — если навсегда?
E4. **Anthropic как single point of failure.** Если SDK сломается, API
подорожает x10, лимиты упадут — как Agenter переживёт?
E5. **Что упустил ты как аудитор.** Какие данные нужны, чтобы дать
полноценную оценку, которых сейчас нет в материалах?
