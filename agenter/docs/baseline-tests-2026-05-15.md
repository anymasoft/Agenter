# Методология baseline-тестирования Agenter после очистки SYSTEM_PROMPT

Дата: 2026-05-15
Версия Agenter: post-cleanup (удалены ~770 строк BSL-примеров из SYSTEM_PROMPT)
Модель: Claude Sonnet 4.6

## 1. Цель и контекст

Из SYSTEM_PROMPT Agenter удалены хардкод-примеры BSL-кода (~770 строк). Цель тестов — выявить РЕАЛЬНЫЕ пробелы в знаниях "голой" LLM при работе с 1С: какие задачи модель решает без подсказок, а где галлюцинирует или не знает синтаксис/API.

Результаты будут использованы как data-driven основание для подключения MCP-серверов в Phase 2 (вместо подключения "на всякий случай"). Если задача проходит без MCP — значит base LLM достаточно, MCP не нужен. Если FAIL — фиксируем конкретный пробел и подбираем MCP, который его закрывает.

В распоряжении Agenter — 20 tools: `db_dump`, `db_load`, `meta_compile`, `meta_edit`, `cfe_borrow`, `cfe_patch_method`, `cfe_validate`, набор `bsl_*` (6 шт), `read_file`, `edit_file`, `write_file`, `list_ext_files`, `clone_object`, `get_sample_object`, `validate_ext_structure`.

## 2. Принципы методологии

1. **Одна задача — одна тема.** Не комбинируем "создай документ + добавь отчёт + настрой роль". Изоляция позволяет точно атрибутировать пробел.
2. **От простой к сложной.** T1 → T5 идёт по нарастающей. Если T1 уже FAIL — нет смысла проводить T5, сначала чиним базу.
3. **Категоризация пробелов.** Каждый зафиксированный FAIL относим к одной из пяти категорий: BSL-синтаксис, платформенный API, БСП-методы, структура метаданных, правила расширений. Это даёт прямой мэппинг "пробел → MCP-сервер".
4. **Не помогаем в процессе.** Никаких подсказок, переформулировок, "попробуй вот так". Наблюдаем как есть — иначе мерим не модель, а помощь оператора.
5. **Чёткий success-criteria.** Для каждой задачи заранее зафиксировано, что значит "получилось". Не "выглядит правильно", а конкретные проверки (`cfe_validate` PASS, `db_load` без ошибок, проведение в 1С отрабатывает).

## 3. Тестовые задачи

### T1 — тривиальная: справочник с двумя реквизитами

**Промпт оператора:**
> Создай в тестовом расширении справочник `пгт_ТестовыйСправочник` с двумя строковыми реквизитами: `Код` (длина 11) и `Наименование` (длина 150).

**Success criteria:**
- Объект создан в `ext_src/Catalogs/пгт_ТестовыйСправочник/`
- `cfe_validate` PASS
- `db_load` без ошибок
- В 1С справочник открывается, реквизиты на месте

**Knowledge gaps to watch:**
- Знает ли формат JSON для `meta_compile` (имена полей, обязательные ключи)
- Помнит ли префикс `пгт_` для объектов расширения
- Понимает ли, что `Код` и `Наименование` — стандартные реквизиты и их не нужно объявлять отдельно

---

### T2 — типовая: документ с ТЧ и движениями

**Промпт оператора:**
> Создай документ `пгт_ТестовоеПеремещение` с табличной частью `Товары` (поля: Номенклатура — ссылка на справочник Номенклатура, Количество — число 15.3, Цена — число 15.2) и движениями по существующему регистру накопления `ТоварыНаСкладах` (расход).

**Success criteria:**
- Документ создан, ТЧ присутствует
- Регистратор `ТоварыНаСкладах` зарегистрирован в `RegisterRecords`
- `cfe_validate` PASS, `db_load` ok
- В 1С документ открывается, ТЧ работает

**Knowledge gaps to watch:**
- Формат `tabularSections` в `meta_compile`
- Формат `registerRecords` (как указывается тип движения)
- Как корректно сослаться на типовой объект: `CatalogRef.Номенклатура`, `DocumentRef.X`
- Знает ли, что `ТоварыНаСкладах` существует в ERP (через `get_sample_object` или `metadatasearch`)

---

### T3 — заимствование: перехватчик ПередЗаписью

**Промпт оператора:**
> Заимствуй типовой документ `РеализацияТоваровУслуг`. Добавь обработчик `ПередЗаписью` в модуль объекта: если реквизит `Контрагент` не заполнен, вызвать исключение с текстом "Контрагент обязателен".

**Success criteria:**
- `cfe_borrow` отрабатывает, объект появляется в `ChangedProperties`/`AdoptedProperties`
- `cfe_patch_method` создаёт корректную аннотацию (`&Перед` / `&После` / `&Вместо`)
- Тело перехватчика — валидный BSL, проходит `bsl_validate`
- После `db_load` в 1С при попытке записи без контрагента появляется исключение

**Knowledge gaps to watch:**
- Структура параметров `cfe_patch_method` (interceptor_type: Before / After / Instead)
- Синтаксис аннотаций `&Перед("ИмяМетода")`, `&После(...)`, `&Вместо(...)`
- Когда нужен `ПродолжитьВызов(...)` (только для `&Вместо`)
- Правильная сигнатура `ПередЗаписью(Отказ, РежимЗаписи, РежимПроведения)`
- Использование `ВызватьИсключение` (а не `throw` или `raise`)

---

### T4 — сложная BSL-логика: проведение с блокировкой остатков

**Промпт оператора:**
> Создай документ `пгт_СписаниеТоваров` с ТЧ `Товары` (Номенклатура, Количество). В модуле объекта реализуй проведение: блокировка остатков по `ТоварыНаСкладах`, проверка достаточности остатка, формирование движений расхода через `НаборЗаписей`. При нехватке — отказ от проведения с понятным сообщением.

**Success criteria:**
- Объект создан, BSL в `ObjectModule.bsl` валиден
- `db_load` без ошибок
- В 1С проведение при наличии остатка — успешно, при нехватке — отказ с сообщением
- Параллельные проведения не дают отрицательного остатка (контроль блокировкой)

**Knowledge gaps to watch:**
- API `БлокировкаДанных`: `Заблокировать()`, элементы `ИсточникДанных`, режим `ИспользоватьИзИсточника`
- Различие `НаборЗаписей` vs `Движения.X.Добавить()` (последнее не подходит для очистки уже записанных)
- Виртуальная таблица `ТоварыНаСкладахОстатки(&МоментВремени, ...)`
- Запрос с `ОБЪЕДИНИТЬ` для контроля остатков по строкам ТЧ
- Параметры процедуры `ОбработкаПроведения(Отказ, Режим)`

---

### T5 — архитектурная: мини-склад от поступления до расхода

**Промпт оператора:**
> Создай складскую подсистему из трёх объектов: документ `пгт_Поступление` (приходует товары на склад), документ `пгт_Расход` (списывает с контролем отрицательных остатков), регистр накопления `пгт_Остатки` (измерения: Номенклатура, Склад; ресурс: Количество). Оба документа делают движения по регистру, расход контролирует отрицательный остаток.

**Success criteria:**
- 3 объекта созданы и валидны
- `cfe_validate` PASS, `db_load` ok
- В 1С: создание поступления → проведение, создание расхода в пределах остатка → ok, расход сверх остатка → отказ
- Отчёт остатков (или ручной запрос в консоли) показывает корректный оборот

**Knowledge gaps to watch:**
- Декомпозиция: способна ли LLM разбить задачу на корректную последовательность tool calls
- Согласование измерений регистра и реквизитов документов
- Повторное использование паттернов из T3/T4 в одной задаче
- Корректное определение тип регистра (Обороты vs Остатки)
- Когда нужны разные виртуальные таблицы: `.Остатки`, `.ОстаткиИОбороты`

## 4. Шаблон протокола замера

Заполнять отдельно для каждой задачи. Хранить в `agenter/docs/runs/T<N>-<дата>.md` либо в одном journal-файле.

```text
ID задачи:           T<N>
Дата / время старт:  YYYY-MM-DD HH:MM
Дата / время финиш:  YYYY-MM-DD HH:MM
Длительность (мин):

Промпт (точная формулировка, скопированная из § 3):
---

Tool calls (в порядке выполнения):
 1. <tool_name>(<краткие параметры>) → OK / FAIL <причина>
 2. ...

Итераций модели:        <N> / <лимит>
Результат:              PASS / FAIL / PARTIAL

Если FAIL / PARTIAL:
  Шаг отказа:           <номер tool call или этап>
  Что LLM не знал:      <конкретно: имя функции / поле JSON / синтаксис / структура>
  Категория проблемы:   BSL syntax | Platform API | BSP | Metadata structure | Workflow logic

Примечания оператора:
---
```

## 5. Категоризация ожидаемых пробелов → MCP-серверы

| Категория пробела            | Признак в логе                                                       | MCP-сервер для Phase 2                |
|------------------------------|----------------------------------------------------------------------|---------------------------------------|
| Платформенный API 1С         | LLM путает имена методов / параметров платформы                      | `alkoleft/mcp-bsl-platform-context`   |
| Методы БСП                   | Галлюцинирует имена общих модулей БСП, не помнит сигнатуры           | `SSLSearchServer`                     |
| Шаблоны паттернов            | Не воспроизводит типовые конструкции (блокировки, движения, формы)   | `1c-templates-mcp`                    |
| Метаданные конкретной базы   | Не знает, какие реквизиты у типового объекта, какие регистры есть    | `BSL Atlas` (уже подключён)           |
| LSP-диагностика BSL          | Синтаксические ошибки проходят валидацию, но падают при `db_load`    | `claude-code-bsl-lsp`                 |

Категория "правила расширений" (`&Вместо` + `ПродолжитьВызов`, ID-конфликты заимствования) — пограничная: частично закрывается templates-mcp, частично — собственным prompt-снипетом Agenter.

## 6. Действия после прогона

После завершения всех 5 задач:

1. **Заполнить сводку по результатам.** Таблица: T1..T5 × PASS/FAIL × категория пробела.
2. **Принципы решения о MCP:**
   - Если задача прошла PASS — соответствующий MCP не нужен (base LLM достаточно).
   - Если FAIL = 0 в категории — MCP этой категории откладываем.
   - Если FAIL > 0 в категории — подключаем MCP в Phase D, далее перепрогон только тех задач, что упали.
3. **Регрессия.** После подключения MCP перепрогон провалившихся задач. Если PASS → MCP оправдан. Если снова FAIL → проблема не в знаниях, а в логике Agenter (промпт, оркестрация, лимиты итераций).
4. **Не подключаем MCP "пакетом".** По одному за раз, с замером дельты.

## 7. Чек-лист подготовки к тесту

Перед запуском каждого прогона убедиться:

- [ ] Agenter запущен и отвечает (проверка health-эндпоинта или CLI ping)
- [ ] База 1С НЕ открыта в Конфигураторе (монопольный режим отключён, иначе `db_load` упадёт)
- [ ] `ext_src/` синхронизирован с БД через `db_dump` (на случай ручных правок пользователя в конфигураторе)
- [ ] В `config.json` Agenter указано тестовое расширение (sandbox-копия с любым префиксом) — на промышленных расширениях прогоны делать нельзя
- [ ] BSL Atlas сервер запущен (`curl http://localhost:8000/health`)
- [ ] Подготовлен пустой файл journal-протокола (`agenter/docs/runs/baseline-<дата>.md`)
- [ ] Зафиксирован baseline-снапшот `ext_src/` (для отката после теста: `Copy-Item ext_src ext_src_pre_T<N>`)
- [ ] Лимит итераций Agenter не занижен искусственно (используем дефолт, иначе мерим не модель, а лимит)

После каждого прогона:

- [ ] Восстановить `ext_src/` из baseline-снапшота
- [ ] Выполнить `db_load` с восстановленным состоянием, чтобы база не накапливала артефакты тестов
- [ ] Заполнить протокол замера (§ 4) до начала следующей задачи
