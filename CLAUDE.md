## КРИТИЧЕСКОЕ ПРАВИЛО: Синхронизация перед началом любой доработки

**Перед началом ЛЮБОЙ задачи, исправления или доработки — первым действием выполнить выгрузку расширения из БД в `ext_src/`.**

Пользователь или другой разработчик мог вносить правки напрямую в конфигураторе 1С. Такие правки живут только в БД и не отражены в `ext_src/`. Если начать работу с `ext_src/` без синхронизации — при последующем `db-load-xml` все ручные правки пользователя будут безвозвратно затёрты.

**Команда синхронизации (выполнять первой, до любого чтения или правки файлов):**

```powershell
& "C:\BUFFER\ERP\.claude\skills\db-dump-xml\scripts\db-dump-xml.ps1" `
    -V8Path "C:\Program Files\1cv8\8.3.27.1859\bin" `
    -InfoBasePath "C:\Users\User\Documents\1C\ERP ProgressTech (демо) 2.5.25.96" `
    -UserName "Администратор (ФедоровБМ)" `
    -Password "" `
    -ConfigDir "C:\BUFFER\ERP\ext_src" `
    -Extension "ИнвентаризацияПГТЕХ" `
    -Mode Full
```

Пользователь не обязан делать дамп самостоятельно — это задача Claude. Отсутствие дампа от пользователя не является причиной пропустить этот шаг.

## КРИТИЧЕСКОЕ ПРАВИЛО: Контроль изменений

**Любое изменение требует явного запроса и явного разрешения.**

- **Перед каждым изменением** — явно описать: что именно будет изменено, в каком файле, в каком месте, почему.
- **Ждать явного подтверждения** от пользователя перед выполнением.
- Запрет распространяется на: BSL-код, XML-метаданные (реквизиты, регистры, перечисления, конфигурация), комментарии, форматирование, переименование.
- **`Documents/ПересчетТоваров/Ext/ObjectModule.bsl` — абсолютный запрет** на любые изменения без явного разрешения. Этот модуль ведётся пользователем самостоятельно.
- Никаких «фоновых» или «заодно» правок — даже очевидных. Каждое изменение = отдельный запрос.

**Rollback-baseline**: `C:\BUFFER\ERP\ext_src_baseline_2026-05-03` — зафиксированное состояние расширения на 2026-05-03. При откате — копировать оттуда в `ext_src`.

## КРИТИЧЕСКОЕ ПРАВИЛО: Экономия токенов

**Перед каждым действием обязательно оценивай расход токенов.**

- Если действие или операция могут повлечь излишний расход токенов (больше, чем необходимо для решения задачи) — **немедленно остановись** и сообщи об этом пользователю с детальным объяснением: почему возможен перерасход и каков его примерный масштаб.
- **Никогда не читай файлы из папки SCHEME напрямую** через инструменты Read/Glob/Grep — только через MCP-инструменты BSL Atlas. Прямое чтение 122 000 файлов = катастрофический перерасход токенов.
- Используй `search_function`, `metadatasearch`, `get_object_details` и другие MCP-инструменты для получения только нужных фрагментов.
- При поиске всегда предпочитай точечный запрос широкому сканированию.

## Проект

1С:ERP 2.5.25 на платформе 8.3.27.1859. Рабочая папка `c:\BUFFER\ERP`.

`SCHEME/` — XML-выгрузка конфигурации из конфигуратора 1С (9 ГБ, 122 000 файлов). Это источник истины для всего кода и метаданных конфигурации. Папка только для чтения — изменения вносятся в 1С, а затем повторно выгружаются.

`ext_src/` — исходники расширения **ИнвентаризацияПГТЕХ**. Все объекты расширения используют префикс `пгт_`. Это единственная папка, куда вносятся изменения кодом.

Параметры подключения к базе данных — в `.v8-project.json` (id `"erp"`).

## Загрузка расширения в БД

Эта команда используется после каждого изменения в `ext_src/`:

```powershell
& "C:\BUFFER\ERP\.claude\skills\db-load-xml\scripts\db-load-xml.ps1" `
    -V8Path "C:\Program Files\1cv8\8.3.27.1859\bin" `
    -InfoBasePath "C:\Users\User\Documents\1C\ERP ProgressTech (демо) 2.5.25.96" `
    -UserName "Администратор (ФедоровБМ)" `
    -Password "" `
    -ConfigDir "C:\BUFFER\ERP\ext_src" `
    -Extension "ИнвентаризацияПГТЕХ" `
    -UpdateDB `
    -StrictLog
```

`-StrictLog` превращает «тихие» предупреждения платформы (отброшенные свойства, неизвестные типы) в ошибку. Всегда используй его. Лог при успехе перечисляет только изменённые объекты метаданных — изменения BSL-кода в лог не попадают, но применяются.

## Архитектура расширения

### Ключевые модули

| Файл | Назначение |
|---|---|
| `CommonModules/пгт_ИнвентаризацияСервер/Ext/Module.bsl` | Вся серверная логика: буфер ТСД, складские акты, формирование дочерних документов |
| `CommonModules/пгт_HTTP_ТСД_Сервер/Ext/Module.bsl` | Серверная логика HTTP API (вызывается из модуля HTTP-сервиса) |
| `HTTPServices/пгт_HTTP_ТСД/Ext/Module.bsl` | Тонкий роутер: делегирует вызовы в пгт_HTTP_ТСД_Сервер |
| `Documents/ПересчетТоваров/Forms/ФормаДокумента/Ext/Form/Module.bsl` | Форма документа: кнопки, доступность элементов |
| `Documents/ПересчетТоваров/Ext/ObjectModule.bsl` | **Ведётся пользователем самостоятельно** — проведение, ПередЗаписью, маппинг статусов. Без явного разрешения не трогать. |

### Буфер ТСД (пгт_БуферТСД)

Регистр сведений, независимый, WriteMode=Independent. Ключ: `Документ + Номенклатура + Характеристика + Серия + Ячейка + УстройствоТСД`. Ресурс: `КоличествоФакт`. Реквизит: `ВремяСканирования`.

Поток данных:
```
POST /результаты → пгт_БуферТСД (без блокировки документа, 180 устройств параллельно)
Кнопка «Перенести буфер» → ПримениБуферКТаблице → КоличествоФакт в ТЧ Товары → очистка буфера
```

`ПримениБуферКТаблице` возвращает количество обновлённых строк. Сопоставление ТЧ с буфером — по ключу `XMLСтрока(Номенклатура)|XMLСтрока(Характеристика)|XMLСтрока(Серия)|XMLСтрока(Ячейка)`. Требование: ТЧ должна быть предварительно заполнена через стандартную кнопку «Заполнить».

### HTTP API (пгт-тсд)

- `GET /задания` — документы ПересчетТоваров со статусом `пгт_Статус = ВРаботе`
- `GET /документ?ссылка=<uuid>` — ТЧ Товары конкретного документа
- `POST /результаты` — запись сканирования в буфер; тело: `{"ссылка", "устройство", "строки": [{"номенклатура", "характеристика", "серия", "ячейка", "количество_факт"}]}`

UUID-строки для ссылок на справочники конвертируются через `XMLЗначение(Тип(...), str)`. Вспомогательная функция `пгт_РазрешитьСсылку` в пгт_HTTP_ТСД_Сервер безопасно обрабатывает null/пустые поля.

### Статусы документа ПересчетТоваров

Собственное перечисление `пгт_СтатусыПересчета` (не стандартное ERP):

```
Подготовлен → В работе → Пересчёт завершён → Утверждён (+ Провести)
```

Маппинг `пгт_Статус` → стандартный `Статус` ERP делается в ObjectModule.bsl (`ПередЗаписью`). Именно стандартный `Статус` управляет редактируемостью `КоличествоФакт` в форме — это поведение платформы, не наш код.

### Паттерны BSL в расширении

**`&Вместо` + `ПродолжитьВызов` первым** = стандартный код выполняется «до», наш код — «после»:
```bsl
&Вместо("ПриСозданииНаСервере")
Процедура пгт_ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)
    ПродолжитьВызов(Отказ, СтандартнаяОбработка); // стандарт первым
    // наш код...
КонецПроцедуры
```

**Очистка записей регистра через НаборЗаписей** (не `Движения.*.Очистить()` — он не удаляет уже записанные в БД данные):
```bsl
НаборЗаписей = РегистрыНакопления.ТоварыНаСкладах.СоздатьНаборЗаписей();
НаборЗаписей.Отбор.Регистратор.Установить(Ссылка);
НаборЗаписей.Очистить();
НаборЗаписей.Записать();
```

**Обновление формы после серверной записи документа** — `ЗначениеВРеквизитФормы(ДокОбъект, "Объект")`, где `ДокОбъект` берётся через `РеквизитФормыВЗначение("Объект")` (не `ПолучитьОбъект()` — он читает из БД и теряет несохранённые изменения формы).

## MCP: BSL Atlas

Индекс конфигурации доступен через MCP-сервер BSL Atlas (`http://localhost:8000`).

**Структурные инструменты (SQLite, мгновенно, без токенов на индексацию):**
- `search_function(name)` — найти функцию/процедуру по имени во всех модулях
- `get_module_functions(module_path)` — все функции конкретного модуля
- `get_function_context(function_name)` — граф вызовов (что вызывает + кто вызывает)
- `metadatasearch(query)` — поиск объектов метаданных (справочники, документы, регистры)
- `get_object_details(full_name)` — реквизиты, табличные части, движения регистров объекта
- `code_grep(pattern)` — поиск текста по BSL-коду с контекстом функции

**Семантические инструменты** (codesearch, helpsearch) недоступны — сервер запущен в режиме `fast`.

Проверка доступности сервера: `curl http://localhost:8000/health`

## Инфраструктура BSL Atlas

- Репозиторий: `c:\BUFFER\tools\bsl-atlas\`
- Виртуальное окружение: `c:\BUFFER\tools\bsl-atlas\venv\`

- Лог сервера: `c:\BUFFER\tools\bsl-atlas-data\server.log`

После новой выгрузки конфигурации из 1С: временно установить `SQLITE_AUTO_REBUILD=true` в `.env`, перезапустить сервер (переиндексация ~15 минут), затем вернуть `false`.

## Структура SCHEME

Стандартная XML-выгрузка 1С по типам объектов:

```
SCHEME/
  Configuration.xml       # корневой манифест конфигурации
  Catalogs/               # справочники
  Documents/              # документы
  CommonModules/          # общие модули
  InformationRegisters/   # регистры сведений
  AccumulationRegisters/  # регистры накопления
  AccountingRegisters/    # регистры бухгалтерии
  Reports/                # отчёты
  DataProcessors/         # обработки
  ...
```

Каждый объект — папка с файлами: `*.xml` (метаданные), `Ext/Module.bsl` (код модуля объекта), `Forms/*/Ext/Form.bsl` (код форм).
